# CPClaw 产品设计蓝图

## 1. 产品愿景

CPClaw 是云枢低代码平台的对话式超级智能体。它不是简单问答机器人，而是面向云枢应用的自然语言操作入口：用户用业务语言表达目标，系统负责理解意图、定位应用能力、补齐信息、生成执行计划、请求确认、调用云枢能力并解释结果。

最终体验目标是：用户可以通过对话完成原本需要在云枢应用中手工完成的大部分操作，包括查询、汇总、新增、修改、删除、流程提交、业务 Action、报表查看、附件填单和历史追溯。

## 2. 产品定位

CPClaw 面向两类核心用户：

- 业务用户：通过自然语言查询和操作自己的业务数据，减少理解云枢应用结构、字段和按钮位置的成本。
- 实施/管理员：配置云枢连接、同步应用元数据、维护模型和检索配置、查看执行审计、治理高风险能力。

产品核心价值：

1. 降低云枢应用使用门槛。
2. 将分散的应用、表单、视图和 Action 统一为对话式入口。
3. 通过本地元数据知识图谱和混合检索提升意图定位准确率。
4. 通过确认、审计和脱敏机制保障企业环境下的安全可控。

## 3. 体验原则

- 目标导向：用户描述业务目标，系统转换为云枢应用操作。
- 本地认知：系统只基于已同步到本地库的元数据检索应用能力，不实时连接云枢做匹配。
- 图谱推理：先定位应用，再沿应用下的数据实体、字段、关系、视图操作和表单操作选择路径。
- 不确定就追问：缺少应用、记录、字段、范围、必填项时，不猜测执行。
- 可推荐：用户输入不完整但存在相近功能时，推荐候选应用、实体或操作。
- 可确认：新增、修改、删除、流程提交、附件上传、不可逆 Action 等操作必须展示摘要并等待确认。
- 可解释：返回结果说明匹配依据、查询条件、执行步骤、数据来源和风险边界。
- Markdown 友好：响应支持表格、列表、引用块、步骤、候选卡片和确认摘要。
- 多轮连续：支持“第一条”“刚才那个商机”“给它写跟进”等上下文指代。
- 附件增强：支持发票、图片、PDF、Excel 等附件驱动业务填单。

## 4. 核心用户旅程

### 4.1 查询与汇总

用户提出查询或统计目标，系统从本地元数据知识图谱和混合检索索引中定位应用、实体、字段和关系路径，调用云枢运行态查询接口，返回摘要、表格和条件说明。

### 4.2 新增与填单

用户提出新增业务对象，系统匹配表单和必填字段，抽取用户输入和附件内容，追问缺失信息，生成表单草稿，用户确认后提交到云枢。

### 4.3 修改与删除

用户描述要修改或删除的记录，系统先定位记录，展示修改前后差异或删除影响范围，用户确认后执行。批量或不可逆操作需要更强确认。

### 4.4 流程与 Action

用户要求提交审批、触发按钮或执行业务动作，系统定位视图操作或表单操作，校验前置条件，展示影响范围，确认后调用云枢 API 或浏览器兜底执行。

### 4.5 多轮上下文操作

用户先查询一批记录，系统保存候选记录上下文。后续用户可通过“第一条”“这个客户”“刚才那条商机”等表达继续操作。上下文引用必须可追溯，不能替代风险确认。

### 4.6 单轮多应用/多实体操作

用户在一句话中提出多个目标时，系统拆解为任务图：先执行查询节点，再基于结果执行后续创建、修改或 Action 节点。涉及写操作时集中展示确认摘要。

### 4.7 附件驱动填单

用户上传发票、合同、图片、PDF 或 Excel，并补充业务描述。系统解析附件，抽取结构化信息，匹配云枢表单字段，生成字段映射和附件上传预览，确认后创建记录并上传附件。

### 4.8 系统管理

管理员配置云枢连接、模型 API 地址、API Key、模型名称、思考模式能力、检索中间件连接信息，测试连接并触发元数据同步。

## 5. 信息架构

产品设计文档分为：

- 产品设计蓝图：说明产品目标、体验原则、用户旅程、MVP 边界和演进路径。
- 需求说明：沉淀业务需求、能力范围、安全要求和非目标。
- 产品规划详细设计：拆分页面、功能域、用户角色、交互模式和版本节奏。
- 测试计划与测试用例：验证产品目标是否可执行、可验收。

## 6. MVP 边界

MVP 优先覆盖：

1. Vue 3 对话入口、历史会话和设置页。
2. 云枢连接配置、连接测试和元数据同步。
3. MySQL 保存会话、设置、加密凭据和云枢元数据。
4. 本地元数据检索和查询类 Agent 模拟链路。
5. 模型配置、对话模型选择和思考模式开关。
6. 附件上传与解析流程的基础框架。
7. 高风险操作确认机制和审计记录。
8. Markdown 响应、候选推荐和结果解释。

MVP 不直接追求覆盖所有云枢复杂页面操作；真实写操作、复杂流程、报表深度分析、浏览器兜底和多应用 DAG 可按受控范围逐步打开。

## 7. 产品演进路径

- 第一阶段：完成对话入口、设置、元数据同步、查询链路、审计和安全底座。
- 第二阶段：完善本地知识图谱、混合检索、候选推荐、多轮上下文和附件填单。
- 第三阶段：开放受控写操作、流程提交、Action 执行和浏览器兜底。
- 第四阶段：增强组织级记忆、业务别名治理、权限联动、报表分析和生产运维能力。

## 8. 专家评审沉淀

由于当前外部专家 Agent 无法在工作区创建隔离环境，本轮蓝图按六个专家视角完成内审并沉淀为改进点：

- 产品策略：需要突出“对话式操作入口”而不是“问答助手”，并以用户旅程串起能力边界。
- 会话体验：需要把澄清、候选推荐、确认、结果解释做成一等交互模式。
- 企业落地：需要补充管理员配置、审计、权限、灰度和安全治理路径。
- 技术架构：需要总蓝图统一描述组件关系和技术路径，再拆专项设计。
- Agent 架构：需要明确本地检索、结构化意图、DAG 计划、多轮上下文和 Reflection。
- 安全运维：需要把凭据、日志、Prompt、附件、审计、Git 提交纳入统一安全边界。
