# CPClaw 需求说明

> 本文是产品需求详细设计。整体产品设计思路见 `../00-product-blueprint.md`。

## 1. 背景

CPClaw 是面向云枢低代码平台的超级智能体助手，目标是让用户通过自然语言对话访问和操作云枢应用，体验接近真实员工在浏览器中使用云枢系统。

典型场景：用户输入“查询我的销售订单数据”，CPClaw 需要理解用户意图，从本地已同步的云枢元数据中定位 CRM 应用、销售订单实体、字段和查询能力，按云枢当前账号权限查询数据，返回摘要、表格和执行说明。

产品期望不是简单关键词回复，而是足够智能地理解用户想完成的业务任务。用户可以用自然语言表达查询、分析、统计、新增、修改、流程提交、附件填单等目标；系统需要先判断业务意图和目标对象，结合本地元数据、会话上下文和可用工具决定下一步。若信息不足，系统应主动追问和给出候选方向，而不是只回复“没有理解”。

## 2. 云枢平台理解

云枢低代码平台包含设计态和运行态：

- 设计态：应用开发与配置入口，可获取应用、目录、业务模型、表单、流程、字段说明、业务规则、视图操作按钮和表单操作按钮。
- 运行态：已发布应用的使用入口，用户在此查询、新增、修改、删除数据，查看报表、发起流程和执行业务 Action。

CPClaw 不在用户请求时实时连接云枢做应用能力检索。云枢应用元数据必须先同步到本地库，再基于本地 Metadata Index 和应用级知识图谱做意图匹配、能力定位和执行规划。

## 3. 产品目标

- 提供统一对话式入口，降低业务用户访问云枢应用的门槛。
- 让智能体基于云枢设计态元数据理解应用、实体、字段、关系和操作能力。
- 当前阶段通过本地 Metadata Index 的精确名称、别名、编码、应用路径、全文关键词、复合业务词和 pgvector 语义召回的混合检索，快速找到用户需要操作的应用功能项；向量检索只能增强召回，不能生成或猜测 `schemaCode`。
- 支持多轮对话、新建会话、历史会话保存和查看。
- 支持单轮多应用、多实体、多步骤的任务拆解。
- 支持附件上传和附件驱动填单，例如发票报销、合同录入、Excel 明细导入。
- 支持普通用户配置自己的云枢访问账号和密码、模型 API，用于对话式操作云枢应用。
- 支持管理员配置自己的云枢账号和密码，读取目标云枢环境元数据并在本地初始化构建应用级知识图谱和检索索引。
- 通过 API 优先、浏览器自动化兜底的方式执行用户授权的业务操作。
- 对高风险操作提供预览、确认、审计和脱敏。
- 当用户提出“分析系统中的商机信息”这类表达时，系统应理解为查询商机数据并进行业务分析，而不是要求用户重新说明云枢对象。
- 当用户提出“系统有多少商机？”、“商机一共有几条？”这类明确统计表达时，系统应识别为查询/计数意图，直接匹配商机对象并返回数量，不应再要求用户补充应用或表单。
- 当用户表达不完整或意图不清楚时，系统应说明已理解到的内容，列出缺失信息和可选对象，引导用户补充应用/表单、动作类型、筛选条件或记录引用。

## 4. MVP 功能范围

### 4.1 对话式 UI

- 多轮对话。
- 新建会话。
- 历史会话保存和查看。
- Markdown 消息展示。
- 候选应用/功能推荐。
- 执行计划预览。
- 长耗时查询或分析提交后，聊天区可以立即显示“正在运行”和耗时，但不得预先生成 Observe、Think、Act、Reflect 或任何尚未执行的步骤。只有收到后端真实 `thought/execution` 事件后才追加对应活动，步骤结果到达后在原位置更新。
- 高风险操作确认卡片。
- 模型选择和思考模式开关。
- 附件上传入口。

### 4.2 系统设置

- 云枢基础地址、登录地址、设计态入口、运行态入口配置。
- 云枢账号配置。
- 云枢密码加密保存，禁止明文入库、入日志、入 Git。
- 模型 API 地址、API Key、模型名称配置。
- 模型能力配置：是否支持思考模式、默认是否启用思考模式。
- 检索中间件连接配置。
- 连接测试和配置有效性检查。

### 4.3 云枢架构初始化

- 获取应用、目录、业务模型、表单、流程、字段、视图操作按钮、表单操作按钮和 Action。
- 将元数据标准化后保存到 MySQL。
- 建立应用级知识图谱。
- 生成检索语料并写入 Elasticsearch/OpenSearch。
- 当前使用 PostgreSQL pgvector 做元数据语义召回；后续可扩展 Milvus 做大规模向量检索。

### 4.4 智能体执行

- 采用结构化意图理解、槽位抽取、本地检索、执行计划、工具调用和 Reflection。
- 支持查询、分析、统计、创建、修改、删除、流程、Action、附件填单等业务意图识别。
- 查询类场景优先落地。
- 查询/统计类表达只要业务对象和动作足够明确，应直接执行查询；只有缺少对象、动作或必要条件且可能查错时才进入澄清。
- 分析类场景应先查询运行态业务数据，再调用模型或规则生成结论、关键发现、风险信号和下一步建议。
- 写操作、删除、流程提交、Action、附件上传必须二次确认。
- 支持多轮上下文引用。
- 支持用户输入不完整时追问或推荐候选功能；低置信度时不得猜测执行。

### 4.5 云枢访问

- API 优先。
- Playwright Java 浏览器自动化兜底。
- 云枢实时连接只用于连接测试、元数据同步和经确认的运行态操作。

### 4.6 智能问数与咨询式报告

- 系统应将智能问数作为核心场景，区分“直接指标问题”和“宽泛经营分析问题”。
- 对“系统有多少商机”这类直接问题，先给出准确结果，不强制扩展成冗长报告。
- 对“上半年我的商机情况怎么样”这类宽泛问题，应结合时间、人员和会话上下文分析用户真正关心的业务结果。
- 系统只能沿本地云枢元数据中真实存在的实体、数据项和关联关系扩展业务链路；不能固定假设商机一定关联线索、合同或回款。
- 当真实元数据和数据支持时，报告可覆盖线索进入量、商机转化量、签约量、阶段分布、金额、风险和改进建议。
- 数量、金额、比例和分布由程序基于云枢真实数据计算；大模型负责规划建议、业务解释和语言组织。
- 报告需要图文并茂，至少支持 KPI、条形图、环形图、漏斗图和折线图，并明确期间、人员范围、数据来源、数据质量和置信度。系统必须在输出前根据用户意图、字段语义、业务顺序、数据关系和类别数量选择呈现方式，禁止把图表类型固定绑定到某个接口或标题关键词。
- 有序业务阶段优先使用阶段漏斗；漏斗宽度按当前阶段及后续阶段累计数量计算，同时保留当前阶段存量，并说明它不是严格同批次转化率。只有真实关联已核验且数值按业务顺序单调递减时，跨对象链路才能使用转化漏斗；否则改用条形对比。时间序列使用折线图，排名/金额使用条形图，少量无序构成使用环形图，高基数分类改用条形图。
- 阶段顺序优先读取云枢数据项元数据中的选项顺序；元数据未提供时，按阶段名称的业务语义推断，不能按数量大小排序冒充业务顺序。每张图必须附带简短口径说明。
- 报告下方提供 2 至 4 个与当前结论高度相关的可点击追问；点击后在同一会话继续执行真实查询。
- “我的”必须映射到云枢人员或负责人字段；无法映射时不能返回全员结果冒充个人结果。
- 元数据关系、日期字段或身份不明确时，系统应输出口径警告或发起澄清，不得编造。
- 报告正文、思考摘要和执行步骤均采用流式反馈；流结束但未收到 `final` 时，本轮请求视为失败。
- 跨对象或全量分析超过 120 秒时，基础设施不得提前截断事件流并向用户显示“未收到 final”；系统必须在配置的任务窗口内返回真实 `final`，或通过 SSE `error` 返回明确的云枢连接、权限或执行错误。
- 执行过程必须采用单轴 Agent 活动流展示，不再压缩成固定四步卡片；思考、元数据匹配、云枢 API 调用、请求参数、返回数据、指标计算、回答生成和完成状态按真实发生顺序呈现。
- 前端禁止根据问题关键词或等待秒数预测“正在规划、正在调用接口、正在生成回答”等未来状态；尚未执行的步骤不展示。
- 正式回答必须在 `answer_start` 后由一个或多个 `answer_delta` 增量写入正文，不能等到 `final` 后一次性显示；短规则回答也应至少让出浏览器绘制帧，使增量变化可被用户感知。
- 同一轮回复在实时执行和消息内“思考与执行过程”中必须复用同一份活动数据；完成后写入消息元数据，刷新历史会话仍可恢复。对话页不再保留独立右侧执行详情栏，避免挤占对话主窗口。
- 助手消息的固定信息顺序必须为“思考与执行过程在上、正式回答在下”。执行中过程区保持展开并实时追加真实步骤，正式回答在过程区下方按 `answer_delta` 增量增长；执行完成后过程区不得消失，必须转换为默认折叠、可再次展开的“思考与执行过程”摘要，历史会话恢复后仍保持该顺序和折叠状态。
- 工具调用结果必须展示业务对象、真实 `schemaCode`、接口地址、返回总数、返回记录数和数据口径；复杂请求/返回数据允许展开查看，不得只保留截断摘要。
- 对话 UI 以用户提供的 `chat-ui/prototype.html` 为布局与交互基准，页面严格采用左侧会话侧栏 + 右侧对话主区两部分，不设置顶部全局导航和右侧执行详情栏。消息内保留可折叠的思考与执行过程，输入框在桌面居中约 800px，移动端自适应且不得产生水平滚动。

### 4.7 对话 UI v2 原型确认稿

- 原型文件：`design/chat-ui-v2-prototype.html`。
- 预览截图：`design/chat-ui-v2-desktop.png`、`design/chat-ui-v2-mobile.png`。
- 页面只允许两个一级布局区域：左侧侧栏与右侧对话主区，不再设置顶部全局导航、顶部模型配置区或右侧执行详情区。
- 左侧依次放置系统 Logo、侧栏折叠、新建对话、历史对话、用户身份和系统设置；当前会话用低对比底色和细边框高亮，删除按钮仅在 hover/focus 时显示。
- 历史对话固定按后端 `updatedAt` 倒序展示。点击会话只改变选中高亮和右侧内容，不得改变列表顺序；新建会话可插入顶部，发送新消息成功后才按新的 `updatedAt` 重新排序。
- 右侧顶部左侧只显示当前会话名称，右侧只显示 Token 消耗和主题切换；主题支持浅色和深色，默认浅色，并记忆用户最近一次选择。
- 右侧主体为对话内容；空状态保留 3 个高频智能问数入口，进入对话后由消息流替代空状态。
- 输入框采用固定底部浮层，桌面宽度约 800px，移动端宽度随屏幕收缩；模型选择、思考深度、附件上传和发送全部位于输入框内部，不提供“新建会话”按钮。
- 右侧独立执行详情栏取消；后续所有执行过程只在消息内部的“思考与执行过程”中展示。
- 视觉规范采用低对比背景、1px 边界和克制阴影；普通卡片圆角不超过 8px，输入框作为工具表面允许使用 14px 圆角。所有图标按钮必须提供可访问名称与悬停提示。
- 左右顶部工具栏必须使用同一高度变量并保持底部分割线精确对齐，确认稿高度为 56px；左侧用户身份区高度为 60px，Logo、头像和内边距采用紧凑尺寸，避免形成厚重的上下框体。
- UI v2 原型支持两种确认状态：默认地址直接展示有内容的多轮对话状态，地址追加 `#empty` 才展示空会话状态。对话态必须包含右侧用户消息、左侧助手回复、完成后默认折叠的思考与执行摘要、可展开的真实步骤、开放排版的正式回答、必要的指标与图表、消息操作按钮和高相关追问。
- 多轮示例需要体现上下文继承：后续短问不得重新要求用户指定实体或时间范围，过程摘要中应明确显示继承了上一轮的对象和筛选口径。
- 2026-07-16 正式页面已按确认稿落地：聊天路由隐藏全局顶部导航，其他管理路由继续保留原导航；左侧系统设置进入管理页面后仍可访问账号与模型、元数据和审计。现有布尔思考能力在 UI 中映射为“快速回答 / 深度思考”两档。
- Token 消耗必须来自模型服务商返回的真实 usage，不得由前端按字符数估算或使用原型静态数字。顶部按当前会话累计展示总 Token，悬停时展示输入、输出与合计明细；明确未调用模型的轮次计为 `Token 0`，历史消息没有 usage 数据时显示 `Token --`。在模型配置未提供上下文窗口上限前，不展示伪造的“已用/上限”比例。
- 输入框工具栏精修规范：模型与思考选择器必须位于同一轻量控制组中，单个控件高度 28px，外层控制组高度 36px；附件与发送按钮统一为 32px。输入框默认使用中性边界，只在聚焦时显示主色；发送中的加载图标不得使用尺寸突兀的组件默认动画。
- 同一视口下所有助手消息使用一致的内容列宽度，短回答不得按内容收缩“思考与执行过程”；执行中的展开面板、执行完成后的折叠面板和最终回答正文必须使用同一宽度。助手正文列固定占满消息容器扣除头像与间距后的可用宽度，过程面板显式占满正文列。用户消息仍按内容自适应，并保留最大宽度约束。
- 当思考步骤、执行结果、图表或正式回答产生增量内容时，浏览器页面滚动条必须自动跟随到最新内容；不得为消息区新增独立滚动条。用户主动向上滚动阅读历史内容后应暂停自动跟随，回到底部附近后自动恢复，避免流式更新抢夺阅读位置。
- 用户发送问题后，输入框内的发送按钮必须在原位置切换为中止按钮，使用方形停止图标并保持按钮尺寸不变。只有用户明确点击中止按钮且后端取消接口确认成功，才允许进入“用户已中止”状态；页面刷新、组件卸载、路由切换、SSE 完成或网络断开均不得冒充用户中止。点击中止后，系统应立即停止前端流读取并通知后端取消当前执行；保留用户问题和已经真实发生的执行步骤，把运行中步骤收束为“已中止”，不展示未经最终校验的部分业务结论，不弹错误提示。中止完成后输入框恢复可用，用户可立即发送下一条问题。
- 每条助手回答底部提供轻量工具栏：只保留复制回答，以及该条回答真实的输出 Token 和总 Token；思考与执行过程通过上方整条摘要直接展开/收起，底部不得再设置重复按钮。Token 紧跟复制图标，以无边框、无背景的低对比内联文字展示，不得单独推到工具栏最右侧或包裹成胶囊；中止消息显示“已中止”，无 usage 数据时不估算 Token。用户消息继续只保留与其相关的轻量操作。

## 5. 非 MVP 范围

- 完整多租户权限体系。
- 企业级统一身份认证和细粒度角色中心。
- 全量流程审批复杂编排。
- 报表大屏和复杂 BI。
- 插件市场。
- 大规模异步任务调度平台。
- 全量生产级浏览器自动化适配。

## 6. 安全要求

- 不保存、不输出用户在对话中提供的任何明文密码。
- GitHub、云枢、模型 API Key、数据库密码、检索中间件密码等凭据必须通过环境变量、系统设置加密存储或安全密钥管理传入。
- 删除、批量修改、流程提交、不可逆 Action、附件上传等高风险操作必须用户确认。
- 所有执行记录、工具输入输出、模型调用摘要和错误信息需脱敏。
- 不把密码、Token、Cookie、API Key、附件敏感原文写入 Prompt。
- 不绕过云枢现有权限体系。
