# MVP 测试用例

## TC-001 前端启动

- 前置条件：已安装前端依赖。
- 步骤：
  1. 进入 `web/`。
  2. 启动 Vue 3 开发服务。
  3. 打开前端页面。
- 预期结果：
  - 页面可访问。
  - 无启动错误。
  - 可看到聊天入口、会话列表入口和设置入口。

## TC-002 后端启动

- 前置条件：已配置 Java 21 和后端依赖。
- 步骤：
  1. 进入 `server/`。
  2. 启动 Spring Boot 应用。
  3. 调用健康检查接口。
- 预期结果：
  - Spring Boot 应用启动成功。
  - 健康检查返回成功状态。
  - 日志中不出现明文敏感信息。

## TC-003 MySQL 连接

- 前置条件：MySQL 可访问，数据库连接参数通过环境变量提供。
- 步骤：
  1. 启动后端。
  2. 执行数据库连接检查。
  3. 执行 Flyway 迁移。
- 预期结果：
  - 后端可连接 MySQL。
  - 迁移执行成功。
  - 必要表结构创建成功。

## TC-004 系统设置保存

- 前置条件：前后端均已启动。
- 步骤：
  1. 打开设置页。
  2. 填写云枢访问地址和账号。
  3. 填写密码并保存。
  4. 重新打开设置页。
- 预期结果：
  - 设置保存成功。
  - 前端不回显密码。
  - 设置查询接口不返回密码或密文内容。
  - 数据库中只保存加密凭据，不保存明文密码。

## TC-005 会话保存与读取

- 前置条件：前后端均已启动，MySQL 连接正常。
- 步骤：
  1. 新建会话。
  2. 输入一条用户消息。
  3. 等待系统生成回复。
  4. 刷新页面并查看历史会话。
- 预期结果：
  - 会话创建成功。
  - 用户消息和助手消息均保存成功。
  - 历史会话可查看。
  - 会话详情可恢复消息列表。

## TC-006 查询类 Agent 模拟链路

- 前置条件：已存在模拟元数据或已完成云枢架构同步，且本地 Metadata Index 可用。
- 步骤：
  1. 新建会话。
  2. 选择系统设置中已配置的模型。
  3. 如果模型支持思考模式，启用或关闭思考模式。
  4. 输入“查询我的销售订单数据”。
  5. 观察 Agent 执行步骤。
- 预期结果：
  - 意图识别为 `query_data`。
  - 能力匹配只查询本地 Metadata Index，不实时连接云枢检索。
  - 能力匹配到 CRM 应用和销售订单实体。
  - 生成查询计划。
  - 返回自然语言摘要、Markdown 表格、查询条件和匹配原因。
  - 工具输入输出审计记录已脱敏。

## TC-007 云枢连接测试

- 步骤：

## TC-009 高风险操作确认

- 前置条件：Agent 已能识别写操作意图。

- 前置条件：已完成一次设置保存、连接测试或 Agent 工具调用。
- 步骤：
  1. 检查后端日志。
  2. 检查工具调用审计记录。
  3. 检查模型调用摘要。
  4. 检查接口响应。
  5. 检查 Git diff。
- 预期结果：
  - 不存在明文密码、Token、Cookie、Session ID、API Key。

## TC-017 明确商机数量查询

- 前置条件：已完成真实云枢架构同步，Metadata Index 中存在商机相关真实模型，且普通用户云枢账号可查询商机运行态数据。
- 步骤：
  1. 新建会话。
  2. 输入“系统有多少商机？”。
  3. 查看 Agent 识别结果、候选对象和回答内容。
- 预期结果：
  - 意图识别为 `query_data`。
  - 系统将“有多少、多少、？”等识别为计数表达，不把它们当成业务对象关键词。
  - 系统匹配商机相关应用/实体，不要求用户重新说明应用或表单。
  - 系统调用云枢运行态查询商机数量。
  - 系统直接返回总数，不进入 `clarify_intent`，不触发高风险确认。
  - 回答正文只展示业务结论，例如“总计 **237** 条”，不展示 `### 执行过程`。
  - Agent 步骤、审计 `plan_json`、`reflection_json` 或工具调用记录中保留动作、业务对象、命中对象、真实 `schemaCode`、运行态总数、原始数据摘要和结论生成说明。
  - fallback 演示来源不能生成业务答案；不得返回 `system_opportunity` 或本地演示商机样本。
  - 商机查询不能误用客户 schema 或客户原始摘要。

## TC-018 业务化处理链路与审计

- 前置条件：已完成真实云枢架构同步，且普通用户云枢账号可查询商机运行态数据。

  2. 输入“系统有多少商机？”。
  3. 查看 Agent 响应步骤。
  4. 打开审计页或调用审计接口查看本次 Agent Run。

  - 审计 `reflection_json` 包含反思状态、检查项、是否通过和是否需要用户输入。
  - 明确查询类请求反思结果为已完成或可返回结果。
  - 高风险写操作必须生成确认，反思状态为等待确认。

## TC-020 按年客户量分析

- 前置条件：已完成真实云枢架构同步，Metadata Index 中存在客户相关真实模型，且客户运行态数据中存在创建时间、登记时间、年份等可识别时间字段。
- 步骤：
  1. 新建会话。
  2. 输入“每年的客户量情况怎么样？”。
  3. 查看 Agent 识别结果、候选对象、执行步骤和回答内容。
- 预期结果：
  - 意图识别为 `analyze_data`。
  - 系统识别动作是分析，业务对象是客户，分析维度是年份。
  - 系统匹配客户相关应用/实体，不要求用户重新说明云枢对象。
  - 系统先查询客户运行态数据；分析类请求应尽量分页拉取完整数据后再聚合。
  - 系统按年份聚合客户量，返回年度分布、峰值年份、趋势判断和下一步下钻建议。
  - 如果运行态总数在系统保护上限内，系统应分页拉取全量记录后再聚合；例如真实云枢客户 total=8694 时，本次返回应为 8694，而不是只用第一页或 1000 条样本。
  - 回答正文展示按年聚合结论、趋势判断和建议，不展示 `### 执行过程`。
  - 客户真实 `schemaCode`、原始客户数据摘要、运行态 total/returned 和分页策略保留在 Agent 步骤与审计工具调用记录中。
  - fallback 演示来源不能生成年度分析结论。
  - 响应步骤按本轮真实发生的工作动态展示步骤名称、过程和结论，不补造固定四步骤，也不直接暴露内部工程标签。
  - 如果无法识别年份字段，系统说明缺失字段并给出补充方向，不编造年度分布。

## TC-021 商机计数后追问阶段分布

- 前置条件：已完成真实云枢架构同步，Metadata Index 中存在商机相关真实模型，且普通用户云枢账号可查询商机运行态数据。
- 步骤：
  1. 新建会话。
  2. 输入“系统有多少商机？”。
  3. 等待系统返回商机总数。
  4. 在同一会话中输入“都处于什么阶段？”。
  5. 查看 Agent 识别结果、候选对象、执行步骤、审计记录和回答内容。
- 预期结果：
  - 第一轮意图识别为 `query_data`，命中真实商机对象，例如 `商机 / int_bu_oppor`。
  - 第一轮助手消息 metadata 保存 `source=runtime-query`、`entityName=商机`、真实 `schemaCode`、total、returned 和 sourceEndpoint。
  - 第二轮发送后 1 秒内，聊天区应出现 CPClaw 的处理中占位回复；收到后端真实步骤事件后，在该轮回答上方逐步显示处理过程，不能预先伪造固定步骤。
  - 第二轮输入虽然没有再次出现“商机”，但“理解问题”阶段应识别为承接上一轮运行态对象，`steps[0]` 中显示“本轮承接上一轮业务对象”。
  - 第二轮 `effectiveQuery` 应包含上一轮真实 `schemaCode` 和对象名，例如 `int_bu_oppor 商机 都处于什么阶段？`。
  - 第二轮意图识别为 `analyze_data`，候选对象仍为上一轮商机对象，不能误用客户 schema 或其它同名对象。
  - 系统重新调用云枢运行态查询接口，按阶段或状态字段聚合商机数量，返回“按阶段分布”。“都处于什么阶段？”不得被识别成普通明细列表查询。
  - 后端最终返回后，处理中占位回复应被真实助手回答替换，消息内过程区保留真实 Agent 步骤并可恢复。
  - 阶段分布必须来自真实云枢运行态数据，不能复用上一轮总数结论；若需要补详情，应使用受控并发和记录上限，避免无限串行请求。
  - 回答正文只展示业务结论和 Markdown 聚合结果，不展示 `### 执行过程`；`schemaCode`、原始数据摘要和 `effectiveQuery` 保留在 Agent 步骤与审计工具调用记录中。
  - fallback 演示来源不能生成阶段分布结论。
  - 真实商机验证样例：`int_bu_oppor` total=237，阶段分布为未签约 135、暂停 13、已签约 65、停止 20、丢单 4；真实复测中该回答正文约 215 字符，不应返回 237 条明细造成超长消息。若接口耗时过长，应记录为性能优化项，而不能降低准确性。

## TC-022 同名对象追问时继承真实 schemaCode

- 前置条件：已完成真实云枢架构同步，Metadata Index 中存在多个同名或相近“商机”对象，例如 `CRM / 商机 / int_bu_oppor` 与 `项目基础数据 / 商机 / business_opportunity`，且两者均可查询运行态数据。
- 步骤：
  1. 新建会话。
  2. 输入“项目基础数据里的商机有多少？”。
  3. 等待系统返回该应用路径下商机总数。
  4. 在同一会话中输入“分别在什么阶段？”。
  5. 查看第二轮候选对象、`steps[0]`、审计 `plan_json.observe` 和工具调用记录。
- 预期结果：
  - 第一轮通过显式应用路径命中 `项目基础数据 / 商机 / business_opportunity`，而不是默认 CRM 商机。
  - 第二轮“理解问题”阶段继承上一轮运行态对象，并把 `business_opportunity` 写入 `effectiveQuery`。
  - 第二轮仍命中 `business_opportunity` 并查询该对象运行态数据，不能跳回 `int_bu_oppor`。
  - 第二轮返回该对象的阶段/状态分布。
  - 审计记录能够看到继承对象、继承 schemaCode、有效查询、运行态 total/returned 和原始数据摘要。
  - 如果上一轮不是真实运行态查询或没有真实 `schemaCode`，系统不得强行继承对象，应进入澄清或重新匹配。

## TC-023 向量语义召回增强元数据匹配

- 前置条件：已完成真实云枢架构同步；`CPCLAW_VECTOR_ENABLED=true`；PostgreSQL pgvector 可访问；Embedding 配置可用；向量索引已从 `metadata_search_documents` 重建。
- 步骤：
  1. 输入不直接包含标准对象名但语义明确的问题，例如“帮我看看销售机会池子”或“潜在成交项目情况”。
  2. 查看元数据候选、Agent 步骤和审计记录。
- 预期结果：
  - 系统通过向量语义召回补充候选，并命中真实商机对象，例如 `CRM / 商机 / int_bu_oppor`。
  - 最终候选必须来自 MySQL `metadata_search_documents`，不能由向量结果直接生成 `schemaCode`。
  - 匹配原因或审计记录中保留“向量语义召回”、相似度、模型名和融合后分数。
  - 如果向量服务不可用，系统应退回确定性检索，不影响已有明确对象问法。

## TC-024 向量检索不得覆盖确定性高置信命中

- 前置条件：同 TC-023，且元数据中存在多个语义相近对象。
- 步骤：
  1. 输入真实编码，例如 `int_bu_oppor`。
  2. 输入显式应用路径，例如“项目基础数据里的商机”。
  3. 查看候选排序和最终执行对象。
- 预期结果：
  - 编码精确匹配必须优先命中 `int_bu_oppor`，即使向量召回返回其它客户或商机候选。
  - 显式应用路径必须优先命中该路径下的真实对象，例如 `项目基础数据 / 商机 / business_opportunity`。
  - 向量相似度只能作为有限加分，不能覆盖编码精确、名称精确、显式路径和上一轮真实 `schemaCode` 继承。
  - 如果 Top1/Top2 分数接近且无明确路径或编码，系统应进入澄清或候选选择，而不是武断执行。
## TC-025 打开历史回复恢复消息内处理流程

- 前置条件：前后端均已启动；已完成至少一次真实或 mock 云枢查询；会话中存在 CPClaw 助手回复。
- 步骤：
  1. 新建或打开一个会话。
  2. 输入“系统有多少商机？”并等待 CPClaw 返回。
  3. 展开该 CPClaw 回复上方的“思考与执行过程”。
  4. 刷新页面或重新打开历史会话。
  5. 再次点击同一条 CPClaw 回复。
  6. 查看浏览器网络请求和该轮消息内的流程区域。
- 预期结果：
  - 新回复的助手消息 `metadataJson` 必须包含 `agentRunId`；运行态查询回复仍保留 `source=runtime-query`、`entityName`、真实 `schemaCode`、total、returned 和 sourceEndpoint。
  - 刷新或打开历史会话后，点击 CPClaw 回复时，前端应读取 `agentRunId` 并调用 `/api/audit/agent-runs/{id}`。
  - 消息内流程区域应按后端事件展示本轮实际步骤、过程和结论，不得补造固定四步骤。
  - 流程摘要应包含用户目标、有效目标、意图、动作、业务对象、候选对象、候选编码、工具 total/returned、运行态来源和结果校验状态；不直接暴露内部工程阶段标签。
  - 历史消息如果没有 `agentRunId`，但 metadata 中仍有 `source=runtime-query`、`schemaCode`、total 等旧字段，消息内可展示有限运行态摘要。
  - 历史消息如果既没有 `agentRunId` 也没有可恢复 metadata，应提示“这条历史消息没有关联的执行流程”，不能静默无响应。
  - 展开某条 CPClaw 回复只显示该轮流程，不得影响其它消息或会话顺序。

## TC-026 执行过程中主动中止

- 前置条件：前后端均已启动；存在一个能够产生多个思考/执行事件和流式回答的长耗时查询。
- 步骤：
  1. 输入长耗时分析问题并点击发送。
  2. 确认发送按钮原位切换为方形中止按钮，悬停提示“中止执行”。
  3. 在元数据匹配或云枢查询阶段点击中止。
  4. 再次发起查询，在正式回答已经输出部分内容时点击中止。
  5. 重复点击中止，并在中止后刷新会话。
- 预期结果：
  - 中止按钮与发送按钮尺寸、位置一致，不导致输入框布局跳动。
  - 不点击中止按钮时，页面热更新、组件重新挂载、路由切换或 SSE 正常结束不得生成“用户已中止”消息；“用户中止”必须能够追溯到显式取消接口调用。
  - 执行中刷新页面或断开 SSE 后，后端不得调用取消接口、不得持久化用户中止状态，也不得尝试把已提交的事件流响应再次包装为 JSON 错误响应。
  - 构造执行时间超过 120 秒的跨对象分析，事件流应保持到配置的任务窗口，最终收到 `final` 或明确 `error`；不得显示“流式响应已结束，但未收到 final”。连续查询同一云枢环境的多个关联对象时应复用认证会话，不得为每个对象重复完整登录探测。
  - 点击后前端立即停止接收新的 SSE 事件、停止计时和自动滚动，不弹“后端服务不可用”等错误。
  - 前端调用 `/api/conversations/executions/{executionId}/cancel`，后端执行句柄进入取消状态并中断工作线程；取消后不得再发送 `answer_delta`、`answer_end` 或 `final`。
  - 用户问题保留；已发生的真实步骤保留，所有仍为 `running` 的步骤转为 `cancelled`，追加“执行已中止”终态。
  - 未经最终校验的部分正文不作为业务结论展示或持久化；助手显示“本次执行已由用户中止，未生成最终结论”。
  - 重复取消幂等，不生成重复终止记录；完成与取消竞争时只能出现一个终态。取消接口应返回 `cancelled=true` 和取消状态；若执行已进入 `committing/completed`，返回 `cancelled=false`，前端继续接收正常 `final`。
  - 中止完成后输入框恢复可用，可立即发送新问题；刷新历史会话后仍能恢复用户问题、取消提示和执行时间线。

## TC-027 助手回答尾部工具栏

- 前置条件：会话中同时存在正常模型回答、无 usage 的历史回答和已中止回答。
- 步骤：
  1. 查看每条助手回答底部工具栏。
  2. 点击复制按钮并核对剪贴板内容。
  3. 点击上方“思考与执行过程”摘要，观察同一消息的执行过程展开和收起。
  4. 核对回答底部 Token 与顶部会话累计 Token。
  5. 在 375px 移动端视口检查布局。
- 预期结果：
  - 助手工具栏默认低对比常显，只包含复制与 Token 信息；用户消息不展示无关 Token 信息，底部不出现重复的执行过程切换按钮。
  - 点击上方摘要只切换当前消息的执行过程，不改变消息顺序、不创建右侧详情栏。
  - 单条回答显示真实“输出 Token · 总计 Token”，数据来自该消息 `metadataJson.usage`；文字必须紧跟复制图标，无边框、无背景、无胶囊。多条回答之和与顶部会话累计口径一致。
  - 无 usage 的历史回答不显示伪造数字；中止回答显示“已中止”。
  - 移动端工具栏不遮挡正文、不超出消息宽度、不产生页面水平滚动。
