# Graphify 全应用元数据图谱验证

## 1. 验证范围

本轮验证 Graphify v8 兼容图投影是否覆盖本地真实云枢元数据中的全部应用，而不是只覆盖 CRM。验证环境使用本地 MySQL 8.0.36 和已初始化的真实云枢设计态元数据。

## 2. 自动化用例

`CpClawApiTests` 覆盖以下链路：

1. Mock 云枢返回 4 个应用、6 个实体、10 个数据项、1 条实体关系和 2 个 API。
2. `POST /api/metadata/sync` 自动生成 22 个节点和 30 条边，应用覆盖率为 100%。
3. `GET /api/metadata/graph/overview` 返回节点、边分类和逐应用覆盖率。
4. `GET /api/metadata/graph/neighborhood` 返回 CRM 商机的一跳应用、字段、关联实体和 API 能力。
5. `GET /api/metadata/graph/export` 返回 Graphify Node-Link JSON，节点和边数与活动快照一致。
6. `POST /api/metadata/graph/rebuild` 基于相同元数据再次构图，前后节点稳定键和边稳定键集合完全一致。
7. 图谱重建后，原有元数据检索、真实运行态查询、SSE 回答和会话持久化继续通过。

## 3. 真实 MySQL 验证

Flyway V6 首次在真实 MySQL 执行时发现 `utf8mb4` 复合索引长度问题。测试环境 H2 不会暴露该限制。稳定键列由 768 字符调整为 512 字符后，清理失败迁移留下的空表并重新执行，V6 成功应用；V1-V6 全部为成功状态。

真实全应用构图结果：

| --- | ---: |
| 应用覆盖 | 29 / 29 |
| 覆盖率 | 100% |
| 图节点总数 | 18,129 |
| 应用节点 | 29 |
| 实体节点 | 953 |
| 数据项节点 | 17,141 |
| API 节点 | 6 |
| 图边总数 | 25,700 |
| 应用包含实体 | 953 |
| 实体包含数据项 | 17,141 |

其中 944 条实体显式关系和 944 条字段引用均标记为 `EXTRACTED`；5,718 条 API 适用边来自通用 API 的适用对象类型，标记为 `INFERRED`，不会冒充云枢显式关系。

## 4. Graphify 导出验证

实际生成文件：`E:\CPClaw\.tmp\graphify-out\graph.json`。

- 文件大小：25,589,527 字节。
- `directed=true`。
- `multigraph=false`。
- `provider=graphify-v8-compatible`。
- `schema_version=graphify-v8-node-link`。
- `nodes=18129`。
- `links=25700`。

文件可由标准 JSON 解析器完整解析，节点和边数量与 MySQL 当前 `ACTIVE` 快照一致。

第二次重建后数据库保留两个快照。以 `stable_key` 双向比较 `ACTIVE` 与最近 `STALE` 快照：新增或缺失节点为 0，新增或缺失边为 0。

## 5. 邻域质量验证

CRM 商机中心节点使用稳定键 `entity:zlcsstcrm:int_bu_oppor`。两跳验证发现，如果允许通过应用和通用 API 节点继续扩散，会把同应用或全部实体误当成语义近邻。遍历策略已经调整：

- `application` 和 `api_endpoint` 在邻域中可以展示，但作为终止节点，不继续扩散。
- `entity`、`data_item` 和显式引用关系可以继续遍历。
- 默认最大深度 3、最大节点 500；响应返回 `truncated` 和实际候选计数。

该策略保留应用归属和 API 能力的可解释性，同时避免形成无意义的全连接路径。

## 6. 阻断条件

以下任一情况出现时不得标记本轮完成：

- 图应用数不是 29，或覆盖率低于 100%。
- 应用、实体、数据项节点数与权威表不一致。
- 强关系存在悬空目标但未进入未解析计数。
- 相同元数据重建后稳定键变化或产生重复。
- Graphify 文件节点/边数与活动快照不一致。
- 图谱构建失败导致原有元数据检索或真实云枢运行态查询不可用。
- 前端在桌面、平板或移动端出现横向溢出、控件遮挡或错误统计。
