# 智能问数与咨询式报告引擎

- 版本：1.0
- 日期：2026-07-15
- 状态：第一阶段已实现

## 1. 目标与边界

智能问数不是把一张业务表转换成自然语言。系统需要理解用户询问的业务主体、时间范围、人员范围和分析深度，从本地云枢元数据中找到真实实体、数据项、关联关系和查询动作，形成可执行的数据路径，再调用云枢运行态接口获取真实数据。

对于“2026 年上半年我的商机情况怎么样”这类宽泛经营问题，系统应在真实元数据关系支撑下扩展到上游和下游对象，例如线索、合同或回款，计算业务转化链、阶段、金额和风险，并生成图文并茂的咨询式报告。

以下行为禁止：

- 固定规定所有商机问题都必须查询线索或合同。
- 编造不存在的 `AppCode`、`SchemaCode`、字段编码或关联关系。
- 将当前登录账号直接当作云枢人员姓名。
- 身份映射失败时返回全员数据并宣称是“我的”数据。
- 让大模型直接计算必须精确的数量、金额和转化率。
- 从 Markdown 文本反向解析图表数据。

## 2. 核心执行链路

```text
用户问题与会话上下文
  -> 主体、期间、人员范围、分析深度识别
  -> 本地元数据检索定位主业务对象
  -> 读取完整邻接元数据图（实体、字段、关系、API）
  -> 将问题与元数据图交给大模型补充分析规划
  -> 规划结果与确定性规则合并并进行元数据白名单校验
  -> 查询主对象及高相关关联对象的云枢运行态数据
  -> 程序执行时间、人员、数量、金额、阶段、转化和风险计算
  -> 将已核验报告事实交给大模型流式撰写咨询式结论
  -> 返回 Markdown 正文 + KPI + 图表 + 风险 + 可点击关联追问
```

模型承担规划建议和语言表达，程序承担元数据真实性、权限边界、数据查询和指标计算。模型失败或超时时，系统使用已核验指标生成稳定报告，不混用未完成模型内容。

## 3. 报告模式判定

报告模式用于宽泛、多维经营问题，例如：

- `2026 年上半年商机情况怎么样？`
- `全面分析一下当前项目经营情况。`
- `给我一份客户健康度诊断报告。`

明确的单维度问题保留快速精确路径，例如：

- `系统有多少商机？`
- `每年的客户量情况怎么样？`
- `商机金额合计是多少？`
- `各阶段分别有多少商机？`

这样可以避免为了“显得智能”而过度扩展用户明确的问题，也控制多实体查询的响应时间。

## 4. 元数据图规划

主对象确定后，规划器读取：

- 主对象全部数据项。
- 以主对象为来源或目标的关联关系。
- 关联字段 `sourceDataItemId`。
- 相邻对象及其数据项。
- 可用查询 API 动作。

关联对象按以下信号评分：

1. 用户问题是否明确提到相邻对象或关系。
2. 大模型规划是否选择该实体、关系或字段。
3. 主体和相邻实体是否位于可解释的业务生命周期位置。
4. 是否属于同一应用。
5. 是否存在日期、负责人、阶段、金额等可分析字段。

第一阶段最多选择三个高相关相邻对象，防止关系图爆炸和运行态查询失控。关联关系只用于选择和解释业务链路，不能等同于数据库 Join；跨对象指标必须基于真实运行态记录和关联字段计算。

## 5. 字段角色识别

系统不固定使用某个字段编码，而是根据显示名、编码、类型和描述为字段识别角色：

- 日期：创建时间、业务发生日期、签约时间等。
- 人员：负责人、拥有者、客户经理、跟进人等。
- 阶段/状态：销售阶段、业务状态等。
- 金额：预计签约金额、合同金额、项目金额等。
- 关联：关联表单、引用字段及目标实体。

字段选择必须保留字段名称和编码作为报告口径。没有可靠日期字段时，相关对象不能参与指定期间指标；没有可靠人员字段时，相关对象不能参与“我的”指标。

## 6. 时间与身份口径

第一阶段支持：具体年份、上半年、下半年、今年、去年和本月。所有时间过滤在程序侧按元数据识别出的真实日期字段执行。

“我的”范围先使用当前普通云枢账号，在人员字段的结构化值中递归匹配账号、手机号或人员 ID。若不能匹配：

- 不返回全员数据冒充个人数据。
- 报告置信度降低。
- 输出清晰的数据口径警告。
- 关联追问可引导用户补充云枢人员姓名或人员 ID。

后续应从云枢登录会话或人员接口建立账号到用户 ObjectId 的正式映射，替代当前账号字段匹配。

## 7. 确定性指标

第一阶段支持：

- 主对象数量和金额合计。
- 阶段或状态分布。
- 上游对象数量、主对象数量和下游对象关联结果组成的业务链路图。
- 仅在关联表单字段能够提取并匹配业务 ObjectId 时计算关联转化率；不能验证逐条关系时只展示同期规模，不输出推测转化率。
- 签约日期或签约状态识别的签约数量。
- 阶段金额分布或按月新增趋势。
- 阶段过度集中、金额缺失、低转化等风险信号。

指标计算完成后必须进入独立的 `InsightVisualizationPlanner`，由规划器根据指标语义选择图表，而不是由报告服务硬编码类型：

- 有序阶段：`funnel`。阶段顺序优先读取数据项 `rawJson` 中的选项定义，缺失时按线索、确认、需求、方案、招采、谈判、签约、交付、回款等业务语义排序。
- 阶段漏斗：第一序列为“当前阶段及后续阶段”的累计数量，用于形成单调收窄的漏斗；第二序列保留当前阶段存量。图表必须声明该口径不等同于严格同批次转化率。
- 已核验转化链路：仅当关联关系已验证且数值按业务顺序单调递减时使用 `funnel`；非单调或不可直接相除的节点改用 `bar`。
- 时间序列：使用 `line`，按自然时间顺序展示。
- 排名、金额和跨类别比较：使用 `bar`。
- 少量无序构成：使用 `donut`；类别过多时退化为 `bar`。

报告执行时间线新增“规划报告呈现” thought 节点，完成状态记录最终选择的图表标题和类型。模型可提供问题意图与分析维度，但图表类型必须经过确定性规划器校验，避免模型生成不受支持或具有误导性的图形。

智能报告和金额、阶段、年度等聚合问法默认分页拉取完整列表，单对象保护上限为 20,000 条。若字段有效值覆盖数小于对象总数，KPI 必须显示为“已识别金额合计”等受限口径，图表标题和警告同时标注覆盖数。数量、金额和比例由 Java 计算，不由模型估算。

## 8. 输出协议

`AgentResponse` 新增可选 `insightReport`，结构包括：

- `title / subject / periodLabel / scopeLabel / confidence`
- `kpis`
- `charts`：`bar / donut / funnel / line`，每张图同时保存 `semantic` 和 `description`，用于历史恢复、审计和前端口径说明。
- `sections`：核心结论、问题与风险、建议动作
- `relatedQuestions`
- `dataSources`
- `warnings`

同一结构写入助手消息 `metadataJson.insightReport`，并设置 `presentationVersion=1`。刷新历史会话后，前端可直接恢复报告，不依赖重新执行查询，也不从 Markdown 中抽取数值。

## 9. 流式与交互

报告执行期间沿用统一 SSE 协议：

- `thought`：构建业务元数据图、形成跨对象路径、规划报告呈现、生成报告。
- `execution`：逐对象查询、返回数、数据来源、指标计算。
- `answer_start / answer_delta / answer_end`：模型正式回答流。
- `answer_reset`：模型超时或不完整时切换确定性报告。
- `final`：返回持久化消息、完整 Agent 响应和结构化报告。

前端必须收到 `final` 才把一次流式请求判定为成功。模型网关必须收到 `[DONE]` 或有效 `finish_reason` 才承认模型正文完整；连接提前结束时发送 `answer_reset` 并切换规则报告。规则回答统一使用短分片和轻量节奏输出，避免浏览器在同一事件循环中一次性渲染完整正文。关联追问按钮将问题作为同一会话的新用户消息发送，重新执行元数据和运行态查询，不在浏览器中直接过滤旧数据。

咨询式报告的模型输出窗口为 20 秒，`max_tokens=1800`，Prompt 要求正文约 1000 个中文字符并保证结尾完整。`finish_reason=length` 视为截断失败，即使随后收到 `[DONE]` 也必须重置并回退，不能把半截报告持久化为成功答案。

## 10. 当前限制

- 当前是进程内最多 20,000 条记录的多对象聚合，不适合超大业务表。
- 云枢不接受日期运算符时会读取完整列表并本地过滤；超过 20,000 条需要服务端聚合方案。
- 模型规划目前参与对象和关系选择，但运行时过滤、分组、排序仍需经过统一规划校验器后才能完全由 AI 计划驱动。
- 关系链只遍历主对象一跳邻接对象，尚未支持受控的多跳图搜索。
- 当前系统只有全局普通用户账号，没有正式的多用户身份与数据范围模型。
- 完整报告保存于消息 `metadataJson`，尚未建立独立的报告版本和证据表。
- CSS/SVG 图表满足当前对话报告，复杂交互分析仍需专业图表引擎。

这些限制必须在回答中通过数据来源、警告和置信度呈现，不能由大模型掩盖。
