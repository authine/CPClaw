# CPClaw 关键技术策略

## 1. 配置与元数据必须持久化

CPClaw 的系统配置、账号配置、模型配置和元数据初始化结果都必须持久化到数据库中，不能只保存在前端状态、内存、临时会话或 H2 临时库里。

必须持久化的内容包括：

- 管理员元数据环境配置：云枢设计态/管理端地址、管理员账号、管理员凭据引用、搜索引擎配置等。
- 个人云枢账号配置：云枢运行态地址、用户账号、用户凭据引用等。
- 模型配置：模型名称、模型 API 地址、模型 API Key 凭据引用、模型显示名、是否支持思考模式、默认思考模式等。
- 云枢元数据初始化结果：应用、模型、表单、字段、流程、Action、关系、检索文档、同步批次、同步时间等。
- 会话、消息、Agent 执行记录、确认记录和审计记录。

凭据类信息只能加密保存，数据库中保存凭据引用或密文，不允许在日志、文档、前端响应或 Git 中出现明文密码、Token、API Key。

当前使用 H2 内存库只能作为临时验证环境。生产和正式联调必须使用持久化数据库，例如 MySQL。重启服务后，账号配置、模型配置、元数据索引和历史记录都应该仍然存在。

## 2. 核心技术主线

CPClaw 的核心逻辑不是普通聊天，也不是让用户自己说明云枢对象，而是：

```text
用户自然语言
  -> 理解用户意图
  -> 从已同步的云枢元数据中匹配应用、模型、表单、字段或 Action
  -> 将意图拆解为可执行任务
  -> 调用云枢 API 查询或操作运行态数据
  -> 将真实数据交给规则聚合或大模型推理总结
  -> 返回结果、依据和执行过程
```

云枢元数据是 CPClaw 理解用户意图和生成执行计划的知识底座。Agent 必须基于已同步的真实云枢元数据进行对象识别、能力定位和任务编排，不能凭空猜 `schemaCode`，也不能使用本地演示数据冒充真实业务数据。

## 3. 元数据匹配的非向量检索方案

当前阶段先不做向量化语义检索。优先实现确定性强、可解释、可测试的元数据检索能力：

1. 精确名称召回：模型、应用、表单或字段名称与业务对象词精确一致，例如“商机”“客户”“线索”。
2. 别名召回：将业务别名映射到标准业务对象，例如“销售机会/机会/opportunity”映射为“商机”，“潜客/lead”映射为“线索”。
3. 编码召回：用户直接输入或上下文中出现 `schemaCode`、应用编码、字段编码时，优先按编码召回。
4. 应用路径召回：使用 `graphPath` 匹配应用和对象路径，例如 `CRM / 商机`、`CRM / 客户`。
5. 全文关键词召回：从 Metadata Index 的 `searchText` 中召回名称、编码、应用描述、模型描述和操作关键词。
6. 复合业务词召回：对“客户下面的商机”“商机跟进记录”等多对象表达，识别最终目标对象和相邻业务对象，优先召回可执行目标对象。
7. 辅助对象降权：对“管理、分配、变更、转移、统计、报表、查询、测试、test”等辅助表或测试表降权，避免误选非核心对象。

向量检索保留为后续增强项。当前实现不得依赖向量库、Embedding 或不可解释语义相似度来决定核心业务对象。

## 4. 元数据匹配流程

意图识别后的元数据匹配流程为：

```text
结构化意图
  -> 业务词清洗
  -> 非向量多路召回
  -> 统一打分排序
  -> 图谱/路径校验
  -> 字段和动作补全
  -> 置信度判断
  -> 输出可执行任务
```

业务词清洗需要去掉“系统、有多少、情况怎么样、全部、一下、数据、信息”等非对象词，保留真实业务对象词。例如：

- “系统有多少商机？”保留“商机”。
- “每年的客户量情况怎么样？”保留“客户”，并提取维度“年份”。
- “把系统中的线索全部分析一下”保留“线索”，并识别动作为分析。

召回阶段宁可多拿候选，也不能过早截断。排序阶段再综合名称、编码、应用路径、对象类型、别名、辅助对象降权和上下文目标对象进行打分。

## 5. 意图识别后的元数据匹配关键技术方案

意图识别完成后，系统不能直接把用户话术转成云枢接口参数，也不能让大模型凭经验猜 `schemaCode`。正确做法是先形成结构化意图，再用真实云枢 Metadata Index 和应用级知识图谱把意图落到可执行对象上。

### 5.1 输入与输出

输入是结构化意图对象，至少包含：

- `intentType`：查询、统计、分析、新增、修改、删除、提交流程、执行 Action、附件填单等。
- `objectHints`：从用户话术中抽取的业务对象词，例如“商机”“客户”“线索”“机会”。
- `appHints`：用户显式提到的应用、模块、路径或业务域，例如“CRM”“项目基础数据”。
- `codeHints`：用户直接输入的 `schemaCode`、应用编码、字段编码或上下文里的编码。
- `dimensionHints`：分析维度，例如年份、阶段、负责人、金额区间、行业。
- `filters`：筛选条件，例如本月、某负责人、某客户名称、某阶段。
- `recordReferences`：多轮上下文引用，例如“第一条”“刚才那个客户”。
- `riskLevel` 与 `requiresConfirmation`：用于区分读数据任务和写操作任务。

输出是一个可执行任务描述，至少包含：

- 命中的真实云枢应用、实体、表单、字段、关系路径或 Action。
- 真实 `schemaCode` 或可执行能力编码，必须来自已同步 Metadata Index。
- 动作类型：计数、列表查询、全量分页分析、新增、更新、删除、提交、Action 等。
- 查询条件、分页策略、分析维度和字段选择。
- 置信度、匹配依据、候选列表和是否需要澄清。
- 是否需要用户确认；读数据可直接执行，写操作必须确认后执行。

### 5.2 匹配流水线

```text
结构化意图
  -> 业务词规范化
  -> 构造多路检索请求
  -> Metadata Index 候选召回
  -> 应用图谱与关系路径校验
  -> 规则打分与候选排序
  -> 字段、视图、表单、Action 补全
  -> 置信度与风险判定
  -> 生成可执行任务或澄清问题
```

业务词规范化要把口语化表达映射到标准业务对象。例如“机会、销售机会、客户机会、opportunity”归一为“商机”，“潜客、lead”归一为“线索”。同时去除“系统、有多少、情况怎么样、看一下、帮我、全部、数据、信息”等非对象词。

构造多路检索请求时，不能只拿原始用户输入做一次模糊查询。系统需要组合以下检索入口：

- 原始用户输入。
- 清洗后的业务对象词。
- 标准业务对象词和别名。
- 应用提示加业务对象，例如“CRM 商机”“项目基础数据 商机”。
- 编码提示，例如 `int_bu_oppor`、`crm_customer`。
- 上下文引用解析出的应用、实体或字段。

### 5.3 候选召回策略

当前阶段不做向量化语义检索，候选召回只使用可解释的非向量方式：

1. 精确名称召回：对象名称与标准业务词一致时高优先级召回。
2. 别名召回：通过内置业务别名、用户纠错记忆或组织级别名映射召回。
3. 编码召回：直接匹配应用编码、实体编码、字段编码和 `schemaCode`。
4. 应用路径召回：使用 `graphPath` 判断用户是否显式指定应用或路径，例如“CRM 客户”“项目基础数据里的商机”。
5. 全文关键词召回：检索 `searchText` 中的应用描述、模型描述、操作词和业务描述。
6. 复合业务词召回：识别“客户下面的商机”“商机跟进记录”等多对象表达中的最终目标对象。
7. 上下文召回：如果用户说“第一条”“刚才那个”，先解析上一轮可引用记录，再补充实体和记录 ID。

召回阶段宁可多召回，也不能因为数据库前几条命中结果就提前截断。排序阶段再统一打分和降噪，防止真实核心对象被同名管理表、分配表、统计表或测试表挡住。

### 5.4 规则打分与排序

候选排序至少综合以下信号：

- 名称精确匹配高于名称包含匹配。
- 编码精确匹配高于文本关键词匹配。
- 用户显式提到的应用路径优先于默认业务偏置；例如“项目基础数据里的商机”应优先命中该应用下的商机。
- 通用 CRM 问法可保护 CRM 核心主对象；例如“系统有多少商机？”优先命中真实 CRM 商机主对象。
- 复合业务词要识别最终目标；例如“分析客户下面的商机”最终目标是“商机”，客户是关系或上下文线索。
- 辅助对象降权：管理、分配、变更、转移、统计、报表、查询、测试、test 等对象默认降权。
- 记忆和上下文只能作为加权信号，不能覆盖真实元数据图谱校验。

排序输出必须带可审计解释，例如：业务关键词、目标对象、命中 `graphPath`、真实编码、召回依据、分数和降权原因。

### 5.5 图谱校验与任务补全

候选排序后，必须进入图谱校验，不能只看 Top1 名称相似度：

- 实体必须属于已同步云枢应用。
- 字段必须属于命中的实体或可达关系路径上的实体。
- 关联查询必须存在关系路径；路径不唯一时进入澄清。
- Action、视图操作、表单操作必须来自对应应用、实体、视图或表单。
- 写操作必须检查必填字段、记录引用、影响范围和风险等级。
- 查询/分析类任务需要补全分页策略、字段摘要、聚合维度和过滤条件。

示例：“每年的客户量情况怎么样？”应补全为：对象=客户，动作=分析，维度=年份，数据动作=尽量分页拉取客户运行态数据，聚合字段=创建时间/登记时间/年份字段，输出年度分布和趋势判断。

### 5.6 置信度与澄清阈值

系统应根据 Top1 分数、Top1/Top2 分差、候选数量、关系路径唯一性、字段归属、上下文引用是否明确和操作风险计算置信度。

- 高置信读数据任务：直接执行，并展示执行过程。
- 高置信写操作：生成计划和风险摘要，请求用户确认。
- 中置信任务：展示候选对象，让用户选择应用、模型或表单。
- 低置信任务：不执行工具，只澄清缺失信息。

澄清问题必须基于已掌握的信息，而不是笼统说“没理解”。例如系统可以回复：“我理解你想分析线索，但当前元数据里有多个线索相关对象，请选择 CRM 线索还是市场线索。”

### 5.7 执行与审计要求

生成可执行任务后，读数据任务可以直接调用云枢运行态查询接口；新增、修改、删除、提交、Action、附件上传等写操作必须等待用户确认。

每次执行都必须记录：

- 原始用户输入和结构化意图摘要。
- 元数据候选列表、最终命中对象和真实 `schemaCode`。
- 匹配依据、分数、`graphPath` 和降权原因。
- 云枢运行态接口、查询条件、分页参数、total、returned。
- 原始数据摘要和脱敏后的工具输出。
- Reflection 结果：完成、需要用户输入、等待确认或执行失败。

本地 fallback 只能用于连接和流程自测，不能作为业务答案来源。如果运行态来源是 `local-fallback`，必须停止生成业务数量、趋势和分析结论。

## 6. 查询类任务

当用户表达的是查询、统计、汇总、分析、洞察、趋势判断等读数据意图时，如果意图和对象已经明确，CPClaw 应直接执行，不应要求用户重复补充云枢对象。

示例：

```text
用户：系统有多少商机？
```

CPClaw 应理解为：

- 意图：查询/统计。
- 业务对象：商机。
- 执行动作：统计商机数量。
- 元数据匹配：从云枢元数据中找到商机对象。
- API 动作：调用云枢运行态查询接口，获取商机总数。
- 返回结果：商机总数，并展示命中的对象、真实 `schemaCode`、运行态 total、原始数据摘要和结论生成过程。

示例：

```text
用户：把系统中的线索全部分析一下。
```

CPClaw 应理解为：

- 意图：分析。
- 业务对象：线索。
- 执行动作：查询线索数据并分析。
- 元数据匹配：从云枢元数据中找到线索对象。
- API 动作：调用云枢运行态接口获取线索数据。
- 分析动作：将线索数据交给大模型或规则分析模块。
- 返回结果：线索分析结论、关键发现、风险信号和下一步建议。

## 7. 操作类任务

当用户表达的是新增、修改、删除、提交、作废、流程审批、执行 Action、写入跟进记录等会改变云枢数据的意图时，CPClaw 不能直接执行。

操作类任务必须经过以下流程：

```text
理解用户意图
  -> 匹配云枢元数据对象
  -> 生成待执行操作计划
  -> 展示影响范围和风险
  -> 请求用户确认
  -> 用户确认后再调用云枢 API 执行
  -> 记录审计
  -> 返回执行结果
```

例如“给第一条商机写一条跟进记录”，系统需要先确认上一轮第一条记录是否能绑定到具体记录 ID、跟进记录对象或 Action 是否存在、跟进内容是否完整，以及新增操作是否需要用户确认。

## 8. 澄清与意图收敛

如果 CPClaw 无法从云枢元数据中准确匹配用户意图，系统应通过多轮对话引导用户补充信息，直到获得更真实、更准确、可执行的意图。

触发澄清的典型情况包括：

- 用户表达过于模糊，例如“帮我处理一下”“看一下数据”。
- 用户说出了动作，但没有明确业务对象，例如“统计一下数量”。
- 用户说出了对象，但云枢元数据中存在多个相近对象，例如多个“线索”“客户”“商机”。
- Top1 候选对象置信度不够高，或 Top1/Top2 分数接近，直接执行可能查错对象。
- 用户使用了业务简称、口语化表达或公司内部叫法，系统无法稳定映射到云枢模型。
- 用户要求写入、修改、删除、提交等操作，但缺少必要字段或上下文记录。
- 用户引用“第一条”“刚才那个”“这个客户”等上下文，但系统无法绑定到具体记录 ID。

澄清时不能只回复“我没有理解你的云枢对象”，而应该说明：已理解到的内容、当前不确定的地方、从云枢元数据中找到的可能相关对象，以及用户只需要补充哪一个关键信息。

核心原则：能准确匹配时直接执行；不能准确匹配时主动澄清；澄清要基于已同步的云枢元数据给出候选；用户补充后继续执行，不要让任务断掉。

## 9. 思考过程与流式输出

CPClaw 的对话交互应采用流式输出体验，参考 DeepSeek 的交互方式，让用户能看到系统正在理解、检索、执行和总结，而不是长时间等待后一次性返回结果。

流式输出阶段包括：

```text
接收用户问题
  -> 输出正在理解意图
  -> 输出正在匹配云枢元数据
  -> 输出正在生成执行计划
  -> 输出正在调用云枢 API
  -> 输出正在分析真实数据
  -> 流式输出最终回答
```

思考过程展示的是可审计的工程化摘要，不是暴露不可控的内部隐式推理原文。展示内容应包括意图理解、业务对象识别、元数据匹配、执行动作、API 调用过程、数据结果摘要、结果生成方式和 Reflection 检查。

## 10. 大模型的角色

大模型不是用来凭空猜数据，而是用于：

- 理解用户自然语言意图。
- 辅助识别业务对象、动作、维度和筛选条件。
- 将云枢元数据转化为可执行计划。
- 对真实查询结果进行总结、分析、归纳和建议生成。
- 在信息不足时生成更好的澄清问题。

大模型不能替代真实云枢查询。业务数据必须来自云枢 API。

## 11. 总结原则

CPClaw 的关键技术原则可以总结为：

```text
配置要持久化。
元数据是知识底座。
当前先做非向量的确定性检索：精确名称、别名、编码、应用路径、全文关键词和复合业务词召回。
意图要转换为云枢对象和动作。
查询类任务直接执行并返回真实结果。
操作类任务必须确认后执行。
意图不明确或元数据匹配不稳定时，通过对话引导用户补充信息。
所有业务结论必须来自真实云枢运行态数据。
大模型负责理解、规划和总结，不能编造业务事实。
交互上必须支持流式输出和可审计执行过程。
```
