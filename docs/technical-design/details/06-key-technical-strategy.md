# CPClaw 关键技术策略

## 1. 配置与元数据必须持久化

CPClaw 的系统配置、账号配置、模型配置和元数据初始化结果都必须持久化到数据库中，不能只保存在前端状态、内存、临时会话或 H2 临时库里。

必须持久化的内容包括：

- 管理员元数据环境配置：云枢设计态/管理端地址、管理员账号、管理员凭据引用、搜索引擎配置等。
- 个人云枢账号配置：云枢运行态地址、用户账号、用户凭据引用等。
- 模型配置：模型名称、模型 API 地址、模型 API Key 凭据引用、模型显示名、是否支持思考模式、默认思考模式等。
- 云枢元数据初始化结果：应用、模型、表单、字段、流程、Action、关系、检索文档、同步批次、同步时间等。
- 会话、消息、Agent 执行记录、确认记录和审计记录。

凭据类信息只能加密保存，数据库中保存凭据引用或密文，不允许在日志、文档、前端响应或 Git 中出现明文密码、Token、API Key。

当前使用 H2 内存库只能作为临时验证环境。生产和正式联调必须使用持久化数据库，例如 MySQL。重启服务后，账号配置、模型配置、元数据索引和历史记录都应该仍然存在。

## 2. 核心技术主线

CPClaw 的核心逻辑不是普通聊天，也不是让用户自己说明云枢对象，而是：

```text
用户自然语言
  -> 理解用户意图
  -> 从已同步的云枢元数据中匹配应用、模型、表单、字段或 Action
  -> 将意图拆解为可执行任务
  -> 调用云枢 API 查询或操作运行态数据
  -> 将真实数据交给规则聚合或大模型推理总结
  -> 返回业务结果，并在审计记录中保留执行依据
```

云枢元数据是 CPClaw 理解用户意图和生成执行计划的知识底座。Agent 必须基于已同步的真实云枢元数据进行对象识别、能力定位和任务编排，不能凭空猜 `schemaCode`，也不能使用本地演示数据冒充真实业务数据。

## 3. 元数据匹配的混合检索方案

当前阶段在确定性元数据检索基础上接入向量语义检索。确定性检索仍然是主链路，负责保障名称、编码、应用路径、上下文和风险控制的可解释性；向量检索作为增强层，负责补充口语化、同义表达和业务语义相近表达的候选召回。

必须坚持一个原则：向量检索只能召回和加权真实 Metadata Index 中已经存在的对象，不能直接生成或猜测 `schemaCode`。最终可执行对象必须来自已同步的云枢元数据，且通过图谱、路径和置信度校验。

确定性检索能力包括：

1. 精确名称召回：模型、应用、表单或字段名称与业务对象词精确一致，例如“商机”“客户”“线索”。
2. 别名召回：将业务别名映射到标准业务对象，例如“销售机会/机会/opportunity”映射为“商机”，“潜客/lead”映射为“线索”。
3. 编码召回：用户直接输入或上下文中出现 `schemaCode`、应用编码、字段编码时，优先按编码召回。
4. 应用路径召回：使用 `graphPath` 匹配应用和对象路径，例如 `CRM / 商机`、`CRM / 客户`。
5. 全文关键词召回：从 Metadata Index 的 `searchText` 中召回名称、编码、应用描述、模型描述和操作关键词。
6. 复合业务词召回：对“客户下面的商机”“商机跟进记录”等多对象表达，识别最终目标对象和相邻业务对象，优先召回可执行目标对象。
7. 辅助对象降权：对“管理、分配、变更、转移、统计、报表、查询、测试、test”等辅助表或测试表降权，避免误选非核心对象。

向量语义检索能力包括：

1. Embedding 文本来源：优先使用 `metadata_search_documents.embedding_text`，并补充对象名称、编码、`graphPath`、对象类型和 `searchText`，形成可重建的语义索引文本。
2. Embedding 模型：使用 OpenAI-compatible Embedding 接口，模型名通过环境变量配置，当前目标模型为 `text-embedding-v4`。
3. 向量库：使用 PostgreSQL + pgvector，独立于主业务 MySQL 数据源，通过 JDBC 直连，避免扰动现有 JPA、Flyway 和 MySQL 元数据持久化链路。
4. 向量表：`metadata_vector_documents` 以 Metadata Index 文档 ID 为主键，保存对象类型、对象 ID、名称、编码、路径、风险等级、索引文本、模型名、向量维度、同步批次和向量。
5. 同步策略：云枢元数据初始化写入 MySQL Metadata Index 后，容错写入向量库；向量库或 Embedding 服务不可用时，元数据同步和确定性检索不能失败。后续可将向量索引重建改为后台异步任务。
6. 查询策略：用户问题先生成 query embedding，再从 pgvector 召回 TopK 语义候选，与确定性候选按文档 ID 合并后统一排序。
7. 排序策略：名称精确、编码精确、显式应用路径和上下文继承优先级必须高于向量相似度；向量分只能作为补充分，不能覆盖明确的编码或路径命中。

配置必须通过环境变量或安全配置中心提供，不允许把数据库密码、Embedding API Key 写入代码、文档或 Git。推荐配置项：

```text
CPCLAW_VECTOR_ENABLED=true
CPCLAW_VECTOR_JDBC_URL=jdbc:postgresql://localhost:5432/CPClaw
CPCLAW_VECTOR_USERNAME=postgres
CPCLAW_VECTOR_PASSWORD=<从本机环境变量或密钥管理系统提供>
CPCLAW_VECTOR_TOP_K=20
CPCLAW_VECTOR_MIN_SIMILARITY=0.62
CPCLAW_EMBEDDING_BASE_URL=<OpenAI-compatible Embedding Base URL>
CPCLAW_EMBEDDING_MODEL=text-embedding-v4
CPCLAW_EMBEDDING_API_KEY=<从本机环境变量或密钥管理系统提供>
```

## 4. 元数据匹配流程

意图识别后的元数据匹配流程为：

```text
结构化意图
  -> 业务词清洗
  -> 确定性多路召回
  -> 向量语义召回
  -> 统一打分排序
  -> 图谱/路径校验
  -> 字段和动作补全
  -> 置信度判断
  -> 输出可执行任务
```

业务词清洗需要去掉“系统、有多少、情况怎么样、全部、一下、数据、信息”等非对象词，保留真实业务对象词。例如：

- “系统有多少商机？”保留“商机”。
- “每年的客户量情况怎么样？”保留“客户”，并提取维度“年份”。
- “把系统中的线索全部分析一下”保留“线索”，并识别动作为分析。

召回阶段宁可多拿候选，也不能过早截断。排序阶段再综合名称、编码、应用路径、对象类型、别名、辅助对象降权和上下文目标对象进行打分。

## 5. 意图识别后的元数据匹配关键技术方案

意图识别完成后，系统不能直接把用户话术转成云枢接口参数，也不能让大模型凭经验猜 `schemaCode`。正确做法是先形成结构化意图，再用真实云枢 Metadata Index 和应用级知识图谱把意图落到可执行对象上。

### 5.1 输入与输出

输入是结构化意图对象，至少包含：

- `intentType`：查询、统计、分析、新增、修改、删除、提交流程、执行 Action、附件填单等。
- `objectHints`：从用户话术中抽取的业务对象词，例如“商机”“客户”“线索”“机会”。
- `appHints`：用户显式提到的应用、模块、路径或业务域，例如“CRM”“项目基础数据”。
- `codeHints`：用户直接输入的 `schemaCode`、应用编码、字段编码或上下文里的编码。
- `dimensionHints`：分析维度，例如年份、阶段、负责人、金额区间、行业。
- `filters`：筛选条件，例如本月、某负责人、某客户名称、某阶段。
- `recordReferences`：多轮上下文引用，例如“第一条”“刚才那个客户”。
- `riskLevel` 与 `requiresConfirmation`：用于区分读数据任务和写操作任务。

输出是一个可执行任务描述，至少包含：

- 命中的真实云枢应用、实体、表单、字段、关系路径或 Action。
- 真实 `schemaCode` 或可执行能力编码，必须来自已同步 Metadata Index。
- 动作类型：计数、列表查询、全量分页分析、新增、更新、删除、提交、Action 等。
- 查询条件、分页策略、分析维度和字段选择。
- 置信度、匹配依据、候选列表和是否需要澄清。
- 是否需要用户确认；读数据可直接执行，写操作必须确认后执行。

### 5.2 匹配流水线

```text
结构化意图
  -> 业务词规范化
  -> 构造多路检索请求
  -> Metadata Index 候选召回
  -> 应用图谱与关系路径校验
  -> 规则打分与候选排序
  -> 字段、视图、表单、Action 补全
  -> 置信度与风险判定
  -> 生成可执行任务或澄清问题
```

业务词规范化要把口语化表达映射到标准业务对象。例如“机会、销售机会、客户机会、opportunity”归一为“商机”，“潜客、lead”归一为“线索”。同时去除“系统、有多少、情况怎么样、看一下、帮我、全部、数据、信息”等非对象词。

构造多路检索请求时，不能只拿原始用户输入做一次模糊查询。系统需要组合以下检索入口：

- 原始用户输入。
- 清洗后的业务对象词。
- 标准业务对象词和别名。
- 应用提示加业务对象，例如“CRM 商机”“项目基础数据 商机”。
- 编码提示，例如 `int_bu_oppor`、`crm_customer`。
- 上下文引用解析出的应用、实体或字段。

### 5.3 候选召回策略

当前候选召回采用“确定性召回 + 向量语义召回”的混合方式。确定性召回负责精确、可解释和高置信命中；向量召回负责补充“销售机会池子、潜在成交项目、老客户资源沉淀”等不完全包含标准业务词的表达。

确定性召回包括：

1. 精确名称召回：对象名称与标准业务词一致时高优先级召回。
2. 别名召回：通过内置业务别名、用户纠错记忆或组织级别名映射召回。
3. 编码召回：直接匹配应用编码、实体编码、字段编码和 `schemaCode`。
4. 应用路径召回：使用 `graphPath` 判断用户是否显式指定应用或路径，例如“CRM 客户”“项目基础数据里的商机”。
5. 全文关键词召回：检索 `searchText` 中的应用描述、模型描述、操作词和业务描述。
6. 复合业务词召回：识别“客户下面的商机”“商机跟进记录”等多对象表达中的最终目标对象。
7. 上下文召回：如果用户说“第一条”“刚才那个”，先解析上一轮可引用记录，再补充实体和记录 ID。

向量语义召回包括：

1. 查询向量化：把用户原始问题、上下文改写后的 `effectiveQuery` 和关键业务槽位组合成向量查询文本。
2. TopK 召回：从 `metadata_vector_documents` 中按余弦距离召回语义相近文档。
3. 文档合并：向量候选必须用 `document_id` 回到 MySQL `metadata_search_documents`，找不到真实元数据文档的向量结果直接丢弃。
4. 低相似度过滤：低于阈值的向量候选不能触发执行，只能作为弱参考。
5. 排序融合：向量相似度转换为有限加分，低于名称精确、编码精确、显式应用路径和已继承真实 `schemaCode` 的优先级。
6. 可审计解释：匹配原因中记录“向量语义召回”、相似度、模型名和合并后的最终分数，但默认不在普通对话正文中展开。

召回阶段宁可多召回，也不能因为数据库前几条命中结果就提前截断。排序阶段再统一打分和降噪，防止真实核心对象被同名管理表、分配表、统计表或测试表挡住。

### 5.4 规则打分与排序

候选排序至少综合以下信号：

- 名称精确匹配高于名称包含匹配。
- 编码精确匹配高于文本关键词匹配。
- 用户显式提到的应用路径优先于默认业务偏置；例如“项目基础数据里的商机”应优先命中该应用下的商机。
- 通用 CRM 问法可保护 CRM 核心主对象；例如“系统有多少商机？”优先命中真实 CRM 商机主对象。
- 复合业务词要识别最终目标；例如“分析客户下面的商机”最终目标是“商机”，客户是关系或上下文线索。
- 辅助对象降权：管理、分配、变更、转移、统计、报表、查询、测试、test 等对象默认降权。
- 记忆和上下文只能作为加权信号，不能覆盖真实元数据图谱校验。

排序输出必须带可审计解释，例如：业务关键词、目标对象、命中 `graphPath`、真实编码、召回依据、分数和降权原因。

### 5.5 图谱校验与任务补全

候选排序后，必须进入图谱校验，不能只看 Top1 名称相似度：

- 实体必须属于已同步云枢应用。
- 字段必须属于命中的实体或可达关系路径上的实体。
- 关联查询必须存在关系路径；路径不唯一时进入澄清。
- Action、视图操作、表单操作必须来自对应应用、实体、视图或表单。
- 写操作必须检查必填字段、记录引用、影响范围和风险等级。
- 查询/分析类任务需要补全分页策略、字段摘要、聚合维度和过滤条件。

示例：“每年的客户量情况怎么样？”应补全为：对象=客户，动作=分析，维度=年份，数据动作=尽量分页拉取客户运行态数据，聚合字段=创建时间/登记时间/年份字段，输出年度分布和趋势判断。

### 5.6 置信度与澄清阈值

系统应根据 Top1 分数、Top1/Top2 分差、候选数量、关系路径唯一性、字段归属、上下文引用是否明确和操作风险计算置信度。

- 高置信读数据任务：直接执行，默认只向用户展示业务结论；执行过程写入审计和调试数据。
- 高置信写操作：生成计划和风险摘要，请求用户确认。
- 中置信任务：展示候选对象，让用户选择应用、模型或表单。
- 低置信任务：不执行工具，只澄清缺失信息。

澄清问题必须基于已掌握的信息，而不是笼统说“没理解”。例如系统可以回复：“我理解你想分析线索，但当前元数据里有多个线索相关对象，请选择 CRM 线索还是市场线索。”

### 5.7 执行与审计要求

生成可执行任务后，读数据任务可以直接调用云枢运行态查询接口；新增、修改、删除、提交、Action、附件上传等写操作必须等待用户确认。

每次执行都必须记录：

- 原始用户输入和结构化意图摘要。
- 元数据候选列表、最终命中对象和真实 `schemaCode`。
- 匹配依据、分数、`graphPath` 和降权原因。
- 云枢运行态接口、查询条件、分页参数、total、returned。
- 原始数据摘要和脱敏后的工具输出。
- Reflection 结果：完成、需要用户输入、等待确认或执行失败。

默认对话正文不展示 `### 执行过程`、原始数据摘要、接口参数或内部步骤；这些内容通过审计页、Agent Run、工具调用记录和开发调试视图查看。助手正文使用 Markdown 输出业务结论、分析、建议、列表和表格，前端需要进行 Markdown 解析和安全净化。

本地 fallback 只能用于连接和流程自测，不能作为业务答案来源。如果运行态来源是 `local-fallback`，必须停止生成业务数量、趋势和分析结论。

## 6. 查询类任务

当用户表达的是查询、统计、汇总、分析、洞察、趋势判断等读数据意图时，如果意图和对象已经明确，CPClaw 应直接执行，不应要求用户重复补充云枢对象。

示例：

```text
用户：系统有多少商机？
```

CPClaw 应理解为：

- 意图：查询/统计。
- 业务对象：商机。
- 处理方式：统计商机数量。
- 元数据匹配：从云枢元数据中找到商机对象。
- API 动作：调用云枢运行态查询接口，获取商机总数。
- 返回结果：商机总数。命中的对象、真实 `schemaCode`、运行态 total、原始数据摘要和结论生成过程写入审计记录，不默认展示在普通对话正文中。

示例：

```text
用户：把系统中的线索全部分析一下。
```

CPClaw 应理解为：

- 意图：分析。
- 业务对象：线索。
- 处理方式：查询线索数据并分析。
- 元数据匹配：从云枢元数据中找到线索对象。
- API 动作：调用云枢运行态接口获取线索数据。
- 分析动作：将线索数据交给大模型或规则分析模块。
- 返回结果：线索分析结论、关键发现、风险信号和下一步建议。

## 7. 操作类任务

当用户表达的是新增、修改、删除、提交、作废、流程审批、执行 Action、写入跟进记录等会改变云枢数据的意图时，CPClaw 不能直接执行。

操作类任务必须经过以下流程：

```text
理解用户意图
  -> 匹配云枢元数据对象
  -> 生成待执行操作计划
  -> 展示影响范围和风险
  -> 请求用户确认
  -> 用户确认后再调用云枢 API 执行
  -> 记录审计
  -> 返回执行结果
```

例如“给第一条商机写一条跟进记录”，系统需要先确认上一轮第一条记录是否能绑定到具体记录 ID、跟进记录对象或 Action 是否存在、跟进内容是否完整，以及新增操作是否需要用户确认。

## 8. 澄清与意图收敛

如果 CPClaw 无法从云枢元数据中准确匹配用户意图，系统应通过多轮对话引导用户补充信息，直到获得更真实、更准确、可执行的意图。

触发澄清的典型情况包括：

- 用户表达过于模糊，例如“帮我处理一下”“看一下数据”。
- 用户说出了动作，但没有明确业务对象，例如“统计一下数量”。
- 用户说出了对象，但云枢元数据中存在多个相近对象，例如多个“线索”“客户”“商机”。
- Top1 候选对象置信度不够高，或 Top1/Top2 分数接近，直接执行可能查错对象。
- 用户使用了业务简称、口语化表达或公司内部叫法，系统无法稳定映射到云枢模型。
- 用户要求写入、修改、删除、提交等操作，但缺少必要字段或上下文记录。
- 用户引用“第一条”“刚才那个”“这个客户”等上下文，但系统无法绑定到具体记录 ID。

澄清时不能只回复“我没有理解你的云枢对象”，而应该说明：已理解到的内容、当前不确定的地方、从云枢元数据中找到的可能相关对象，以及用户只需要补充哪一个关键信息。

核心原则：能准确匹配时直接执行；不能准确匹配时主动澄清；澄清要基于已同步的云枢元数据给出候选；用户补充后继续执行，不要让任务断掉。

## 9. 多轮上下文继承策略

CPClaw 必须支持同一会话中的连续追问。用户前一轮问“系统有多少商机？”，后一轮问“分别在什么阶段？”时，后一轮应继承上一轮已经命中的“商机”对象，并继续查询真实云枢运行态数据，而不是要求用户重新说明业务对象。

当前 MVP 先落地对象级上下文继承：

1. 真实运行态查询成功后，助手消息 metadata 中保存 `source=runtime-query`、`entityName`、`schemaCode`、`total`、`returnedRecords` 和 `sourceEndpoint`。
2. 下一轮“理解问题”阶段从最近的助手 metadata 中恢复运行态对象。
3. 如果用户输入没有新的业务对象，但包含“分别、阶段、状态、分布、占比、负责人、金额、按、有哪些、多少”等承接式表达，则判定为继承上一轮运行态对象。
4. 系统生成 `effectiveQuery`，并把上一轮真实 `schemaCode` 和对象名补入查询，例如 `int_bu_oppor 商机 分别在什么阶段？`。
5. “规划路径”阶段基于 `effectiveQuery` 重新做意图识别和 Metadata Index 匹配，“查询数据”阶段重新调用云枢运行态接口。
6. 审计记录中保留 `effectiveQuery`、`inheritedRuntimeObject`、`runtimeContextSchema` 和工具调用结果；普通对话正文只展示业务回答。

这里要特别避免两个错误：

- 不能只继承对象名称。多个应用存在同名对象时，只继承“商机”可能让系统从上一轮的“项目基础数据 / 商机 / business_opportunity”跳回默认 CRM 商机 `int_bu_oppor`。
- 不能直接复用上一轮结论。追问阶段、负责人、金额、年度等维度时，必须重新查询真实运行态数据并按新问题聚合或分析。

阶段/状态类追问的运行态策略：优先使用列表接口已经返回的字段进行聚合；如果列表字段无法识别阶段或状态，再按受控并发补齐运行态详情并聚合。详情补齐必须有并发上限和记录上限，避免对超大对象逐条串行请求导致前台长时间无响应。真实验证结果应同时记录准确性和耗时：例如商机 `int_bu_oppor` total=237 的阶段分布已能返回未签约 135、暂停 13、已签约 65、停止 20、丢单 4，但当前仍需要后续通过字段级元数据、列表显示字段配置或云枢聚合接口继续优化性能。

当前尚未完成完整记录级引用，例如“第一条”“刚才那个客户”“给它上传附件”还需要继续实现记录 ID 绑定、行号映射、过期策略和写操作确认。

## 10. 思考过程与流式输出

CPClaw 的对话交互应采用流式输出体验，参考 DeepSeek 的交互方式，让用户能感知系统正在理解、检索、执行和总结，而不是长时间等待后一次性返回结果。

流式输出阶段包括：

```text
接收用户问题
  -> 输出正在理解问题
  -> 输出正在匹配云枢元数据
  -> 输出正在生成执行计划
  -> 输出正在调用云枢 API
  -> 输出正在分析真实数据
  -> 流式输出最终回答
```

流式状态展示的是可审计的业务化进度摘要，不暴露内部工程标签或不可控的隐式推理原文。默认最终回答只保留业务结论；问题理解、业务对象识别、元数据匹配、处理方式、API 调用过程、数据结果摘要和结果生成方式写入审计记录，必要时在调试视图查看。

前端可见的处理链路统一使用“理解问题、规划路径、查询数据、整理回答”四类业务化标题。`Observe / Think / Act / Reflect` 这类内部实现标签只允许出现在技术文档或后端审计结构字段中，不能直接展示给最终用户。

## 11. 大模型的角色

大模型不是用来凭空猜数据，而是用于：

- 理解用户自然语言意图。
- 辅助识别业务对象、动作、维度和筛选条件。
- 将云枢元数据转化为可执行计划。
- 对真实查询结果进行总结、分析、归纳和建议生成。
- 在信息不足时生成更好的澄清问题。

大模型不能替代真实云枢查询。业务数据必须来自云枢 API。

## 12. 总结原则

CPClaw 的关键技术原则可以总结为：

```text
配置要持久化。
元数据是知识底座。
当前采用确定性检索 + 向量语义检索的混合召回：精确名称、别名、编码、应用路径、全文关键词、复合业务词和上下文继承仍是主信号；向量检索只增强口语化和语义相近表达的候选召回。
意图要转换为云枢对象和动作。
查询类任务直接执行并返回真实结果。
操作类任务必须确认后执行。
意图不明确或元数据匹配不稳定时，通过对话引导用户补充信息。
所有业务结论必须来自真实云枢运行态数据。
大模型负责理解、规划和总结，不能编造业务事实。
交互上必须支持 Markdown 结果渲染、流式输出状态和可审计执行记录；普通对话默认不展示执行过程块。
```

## 13. 云枢业务对象 API 接口元数据化

为避免 Agent 在执行阶段只依赖硬编码规则，CPClaw 将云枢业务对象 API 能力也纳入元数据体系，作为与应用、实体、数据项、关联关系同等重要的执行元数据。

### 13.1 元数据对象

新增 `cloudpivot_api_endpoints` 表，记录每个云枢接口的能力边界：

- `apiCode`：接口能力编码，例如 `runtime_query_list`。
- `name`：接口中文名称，例如“查询业务对象数据集合”。
- `method` 与 `path`：HTTP 方法和接口路径。
- `category`：接口分类，例如运行态数据、元数据。
- `operationType`：能力类型，例如集合查询、详情查询、新增修改、删除、元数据字段查询。
- `riskLevel` 与 `requiresConfirmation`：风险等级和是否必须用户确认。
- `inputSchemaJson`：接口输入参数、必填项和业务含义。
- `outputSchemaJson`：接口输出结构、关键字段和兼容字段。
- `dataScope`：该接口能获取或操作的数据范围。
- `applicableObjectType`：适用对象类型，当前主要是业务实体。
- `rawJson`：接口来源、验证状态、候选能力等扩展信息。

这些接口元数据在云枢元数据初始化时写入数据库，并生成 `objectType=api_endpoint` 的 `MetadataSearchDocument`，可被元数据检索、向量索引和执行规划器共同使用。

### 13.2 当前登记的核心接口清单

| apiCode | 名称 | 方法与路径 | operationType | 输入 | 输出 | 数据能力 | 风险策略 |
|---|---|---|---|---|---|---|---|
| `runtime_query_list` | 查询业务对象数据集合 | `POST /api/runtime/query/listSkipQueryListV2` | `query_collection` | `schemaCode`、`queryCode`、`page`、`size`、过滤、排序和扩展选项 | `totalElements`/`total`/`totalCount`、`content`/`records`/`list` | 获取指定业务对象分页集合、总数和轻量字段；如果只返回 ID，需要详情接口补齐 | 读接口，可直接执行 |
| `runtime_form_load_new` | 查询单个业务对象详情 | `GET /api/runtime/form/loadNew` | `query_detail` | `schemaCode`、`objectId` | `data`、`bizObject`、`bizObject.data`、`formData`、`properties` | 获取单条业务对象完整详情，用于“第一条是什么信息”、列表摘要补齐和字段分析补齐 | 读接口，可直接执行 |
| `runtime_form_save` | 新增或修改业务对象详情 | `POST /api/runtime/form/save` | `create_update` | `schemaCode`、更新时的 `objectId`、待写入 `data` | `success`、保存后的对象或记录 ID | 新增或更新业务对象 | 写接口，必须用户确认；当前作为候选能力，真实环境需验证参数 |
| `runtime_form_delete` | 删除单个业务对象详情 | `POST /api/runtime/form/delete` | `delete` | `schemaCode`、`objectId` | `success`、`message` | 删除业务对象记录 | 高风险写接口，必须用户确认；当前作为候选能力 |
| `metadata_bizproperty_list` | 查询业务模型数据项 | `GET /api/app/bizproperty/list` | `metadata_data_item` | `schemaCode`、`isPublish` | `dataItems`、`properties` | 获取业务对象字段、数据项、关联表单字段，用于构建实体字段和关联关系 | 元数据读接口 |
| `metadata_get_biz_schema` | 查询业务模型 Schema | `GET /api/runtime/form/get_biz_schema` | `metadata_schema` | `schemaCode` | `data`、`dataItems` | 获取业务模型定义和可能包含的数据项结构，作为元数据同步兜底 | 元数据读接口 |

### 13.3 执行规划规则

Agent 在识别用户意图后，不只判断业务对象，还需要结合接口元数据生成可执行路径：

1. 先通过元数据检索定位用户目标：应用、业务实体、数据项或关联关系。
2. 如果命中数据项或关联关系，先回溯到所属业务实体，得到真实 `schemaCode`。
3. 根据用户动作选择接口能力：
   - 统计数量、列表、分析：优先 `query_collection`。
   - 第一条、详情、明细、是什么信息：组合 `query_collection` + `query_detail`。
   - 新增、修改、保存、写入：选择 `create_update`，进入确认流程。
   - 删除、移除、作废：选择 `delete`，进入确认流程。
4. 执行计划中记录：执行对象、`schemaCode`、字段线索、关联线索、接口线索。
5. 查询类可直接调用云枢运行态接口；写操作必须展示影响范围和风险，用户确认后才执行。
6. 右侧处理流程和审计记录展示的是工程化执行摘要，包括接口线索和数据来源，不在普通回答正文中展示冗长执行过程。

### 13.4 对智能体能力的影响

接口元数据化后，CPClaw 的主线从“规则猜接口”升级为“元数据驱动执行”：用户自然语言 -> 意图与对象识别 -> 云枢元数据匹配 -> 接口能力匹配 -> 运行态 API 调用 -> 大模型总结。这样可以解释为什么某个问题应该查集合、为什么追问第一条信息要先取集合再取详情、为什么写操作不能直接执行。

### 13.5 元数据模型浏览页面

云枢元数据页面需要直接展示当前已同步的模型知识，不应只提供应用列表和搜索结果。页面展示对象包括：

- 云枢应用与应用下的实体对象。
- 实体对象中的数据项，包括字段编码、字段名称、字段类型、是否必填、是否关联字段、关联目标实体等。
- 实体对象之间的关联关系，包括来源实体、来源数据项、目标实体、关系类型和关系名称。
- 云枢业务对象 API 动作，包括接口编码、方法、路径、操作类型、输入输出结构、数据能力、风险等级和是否需要确认。

后端通过 `GET /api/metadata/model` 返回聚合后的模型树：应用 -> 实体 -> 数据项 / 关联关系 / API 动作，同时返回全局 API 动作清单。前端元数据页基于该接口提供实体列表、应用筛选、实体搜索和右侧详情 Tab，便于产品、技术和测试人员核对智能体实际掌握的云枢知识。

该页面的目标不是替代 Agent 执行，而是让用户能够验证 Agent 的知识底座是否真实完整：当用户问“系统有多少商机”时，系统命中的实体编码、可用字段、关联客户关系和可调用 API 动作，都应该能在该页面中找到对应依据。
### 13.6 业务对象级 API 动作生成规则

云枢每个业务对象都有独立的运行态业务规则 API。CPClaw 在元数据模型展示和 Agent 执行规划中，不能只展示全局通用接口，而应根据业务对象所属应用编码 `AppCode`、业务对象编码 `SchemaCode` 和运行态记录 `bizObjectId` 生成对象级 API 动作。

每个业务对象固定生成 5 个 API 动作：

| 动作 | 路径规则 | 说明 | 风险策略 |
|---|---|---|---|
| 数据新增 | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Create` | 新增一条业务对象数据 | 写操作，必须用户确认 |
| 数据删除 | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Delete/{bizObjectId}` | 删除指定业务对象记录 | 高风险写操作，必须用户确认 |
| 加载列表数据 | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/GetList` | 查询业务对象数据集合、分页列表和总数 | 读操作，可直接执行 |
| 数据更新 | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Update` | 更新指定业务对象记录 | 写操作，必须用户确认 |
| 加载表单详情 | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Load/{bizObjectId}` | 查询单条业务对象详情 | 读操作，可直接执行 |

其中 `{AppCode}` 必须替换为业务对象所在应用编码，`{SchemaCode}` 必须替换为业务对象编码，`{bizObjectId}` 在删除和详情加载时由运行态列表查询或用户上下文提供。例如综合办公管理应用 `OA` 下的会议申请对象 `Meeting`，列表查询路径为 `/api/api/runtime/business_rule/OA/Meeting/GetList`，详情查询路径为 `/api/api/runtime/business_rule/OA/Meeting/Load/{bizObjectId}`。

当前实现中，`cloudpivot_api_endpoints` 保留为接口能力模板和检索资料；`GET /api/metadata/model` 在返回每个实体对象时，按实体所属应用和实体编码动态生成该实体专属的 5 个 API 动作。这样既能在元数据页面直观看到每个业务对象的真实接口路径，也能让 Agent 在执行规划时把用户意图落到具体业务对象的具体 API 动作上。
### 13.7 元数据页面应用层级展示

云枢元数据页面的模型浏览入口采用“应用 -> 实体对象 -> 数据项 / 关联关系 / API 动作”的层级结构。应用是一级容器，实体对象必须挂在所属应用下面展示，不能只在左侧平铺实体再通过筛选器间接区分应用。

该结构用于解决两个问题：第一，云枢中不同应用可能存在同名实体，用户和测试人员必须先看到实体所属应用，才能判断 `AppCode + SchemaCode` 是否正确；第二，对象级 API 动作路径依赖应用编码和实体编码，页面按应用分组后，可以直接验证每个实体生成的业务规则接口是否替换了正确的 `AppCode` 和 `SchemaCode`。
### 13.8 业务对象 API 的 HTTP Method 规则

业务对象级 API 动作不能统一按 `POST` 展示或执行，必须按云枢接口定义保留真实 HTTP Method：

| 动作 | Method | 路径规则 |
|---|---|---|
| 数据新增 Create | `POST` | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Create` |
| 数据删除 Delete | `DELETE` | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Delete/{bizObjectId}` |
| 加载列表数据 GetList | `POST` | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/GetList` |
| 数据更新 Update | `PUT` | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Update` |
| 加载表单详情 Load | `GET` | `/api/api/runtime/business_rule/{AppCode}/{SchemaCode}/Load/{bizObjectId}` |

流程生效 `Available` 和流程作废 `Cancel` 属于流程类动作，当前先作为后续扩展能力记录，待执行规划支持流程动作时再纳入对象级 API 动作清单。

## 14. 多轮上下文与智能意图识别优化

CPClaw 的多轮对话不能把每一轮都当成新的孤立问题处理。用户在同一会话中说“进行中的项目有多少，项目金额有多少”“第一个是什么信息”“这些分别在什么阶段”时，系统应优先继承上一轮已经查询到的真实云枢运行态对象，并把短追问转成对同一业务对象的继续查询或分析。

当前优化规则如下：

1. 助手每次完成运行态查询后，会在消息 `metadataJson` 中记录 `entityName`、`schemaCode`、`total`、`returnedRecords` 和 `sourceEndpoint`。
2. 下一轮用户输入如果命中追问特征词，例如“这些、上述、刚才、上一条、这个、第一条、第一个、详情、阶段、状态、进行中、金额、合计、汇总、分析、趋势、情况”等，则优先继承上一轮 `schemaCode`。
3. 如果用户在追问中显式提到了新的业务对象，且该对象与上一轮对象不一致，则不继承，重新走元数据检索，避免把“客户下面的商机”误当成上一轮客户继续查询。
4. 继承成功后，Think 阶段直接生成继承对象的 `MetadataSearchResult`，跳过重复元数据召回，减少思考耗时。
5. 对“状态筛选 + 数量/金额汇总”类问题，例如“进行中的项目有多少，项目金额有多少”，统一识别为 `analyze_data`，而不是普通 `query_data`，因为这类问题需要筛选、聚合和解释。
6. 运行态查询继续基于真实云枢接口返回数据；如果只是数量类查询会轻量取数，如果涉及状态、金额、负责人、时间等维度聚合，则按需拉取更多样本或全量上限内的数据进行分析。

该策略的目标是让 CPClaw 在多轮对话中形成“会话级业务对象记忆”：第一轮确定对象，后续短追问优先围绕该对象继续执行，只有当用户明确切换对象或上下文不足时才重新澄清。

## 15. 混合意图规划与流式输出策略

用户期望 CPClaw 既要更智能，也要响应更快。因此意图理解不能简单地“全部交给大模型”，否则每轮都会引入额外等待；也不能只靠关键词规则，否则复杂追问、宽泛分析和上下文指代会显得死板。当前采用“确定性快路径 + 模型规划兜底”的混合策略。

### 15.1 意图规划分层

1. **确定性快路径**：明确计数、详情、金额、阶段、状态、负责人、省份、新老客户等常见查询和聚合，优先由规则层直接识别；如果上一轮已有真实运行态对象，短追问直接继承 `schemaCode`，跳过重复元数据召回。
2. **模型规划兜底**：当问题是短追问、低置信、宽泛分析、复杂指标组合，或无法直接匹配云枢对象时，调用 `ModelGateway.planIntent(...)` 输出结构化意图 JSON，包括动作、业务对象、维度、筛选、字段线索、关联线索、是否需要澄清和置信度。
3. **元数据校验**：模型规划不能编造 `schemaCode`、不能覆盖真实 Metadata Index 的执行对象、不能把写操作降级为查询。最终执行对象必须来自云枢元数据或上一轮真实运行态上下文。
4. **槽位合并策略**：规则层已明确识别出的高质量槽位优先保留，例如“新老客户”“金额汇总”“阶段/状态”；模型只补全空白或弱槽位，例如“无明确维度”“无明确筛选条件”。
5. **澄清策略**：只有对象、动作或必要上下文无法确定时才澄清；查询类和分析类在对象明确时直接执行，新增、修改、删除仍进入确认流程。

### 15.2 流式输出策略

普通 `POST /api/conversations/messages` 仍保留为兼容接口，但前端对话主链路优先使用 `POST /api/conversations/messages/stream`。该接口使用 SSE 返回事件：

- `progress`：兼容旧版进度事件。
- `thought`：可审计的思维步骤摘要，包括理解问题、规划路径和结果校验；不输出模型内部逐字推理。
- `execution`：实际执行动作及结果，包括元数据命中、云枢接口调用、返回总数、返回记录数、数据来源和模型总结状态。
- `answer_start`：正式回答开始，标记回答来源为 `model`、`rule`、`fallback` 或 `direct`。
- `answer_delta`：正式回答增量。模型总结时直接转发 OpenAI-compatible `stream=true` 返回的 token；规则回答按可读片段增量发送。
- `answer_reset`：模型输出不完整或超时时，通知前端清空不完整正文并切换到规则兜底。
- `answer_end`：正式回答流结束。
- `final`：完整 `AgentResponse`，包含持久化后的助手消息、Agent Run、候选对象、执行步骤、耗时和确认信息。
- `error`：异常信息，用于前端恢复输入和提示用户。

事件必须按实际发生顺序发送，不允许在任务结束后模拟进度。典型顺序为：`thought(理解)` → `thought(规划)` → `execution(元数据匹配)` → `execution(云枢查询开始)` → `execution(云枢数据返回)` → `execution(模型总结开始)` → `answer_start` → 多个 `answer_delta` → `answer_end` → `final`。

前端等待态必须是纯事件驱动：创建临时助手消息时正文和步骤数组均为空，只显示运行状态与耗时；不得根据“金额、阶段”等关键词预设执行计划，也不得按等待秒数自动切换到尚未发生的阶段。`thought/execution` 事件到达后才新增步骤，同一动作的 `running -> completed/fallback/needs_input` 更新同一节点。云枢接口拿到响应后必须先把“调用云枢运行态接口”更新为完成态，再新增“云枢数据返回”结果节点。

模型网关已通过 OpenAI-compatible `stream=true` 读取 `choices[0].delta.content`，每个 token 到达后立即发送 `answer_delta`。若模型在硬超时内没有完整结束，系统发送 `answer_reset` 和 fallback 执行事件，再流式输出规则总结，避免把不完整模型正文与兜底答案拼接在一起。

前端将思维步骤和正式回答分区展示，并固定采用“过程在上、正文在下”的消息结构：pending 状态渲染展开的实时活动时间线，Markdown 正式回答容器紧随其后接收 `answer_delta`；收到 `final` 后，实时时间线替换为默认关闭的原生 `details` 折叠区，但不从消息中删除，Markdown 正文仍保持在其下方。历史消息从 `metadataJson.executionTimeline` 恢复时使用同一结构，避免刷新后丢失过程或改变消息顺序。`final` 只负责用数据库持久化后的助手消息和完整审计信息替换临时消息，不再重新播放正文。

SSE 客户端必须串行消费事件回调，保证 `answer_delta` 的处理顺序。每个增量写入后至少等待一次 Vue DOM 更新；对于本地规则或 fallback 的快速片段流，允许让出一个短浏览器绘制帧，避免多个增量在同一渲染批次中被合并成一次性显示。模型 `stream=true` 返回的 token 不增加人工延迟。

模型规划输出的运行态过滤条件必须做问题依据校验。负责人字段只有在用户问题明确出现可验证人员名称时才可执行；“现在、当前、系统、全部、所有”等上下文词不得作为负责人或普通过滤值。`AgentOrchestrator` 与 `CloudPivotRuntimeService` 的兼容规则解析必须使用同一约束，避免规划层无筛选而运行态层再次错误添加人员过滤。

### 15.3 性能边界

- 明确对象继承 + 明确指标聚合的追问不强制调用模型规划，保证第二轮追问尽量走毫秒级 Think。
- 模型规划只作为低置信或复杂意图补强，不能成为每轮固定前置步骤。
- 运行态查询仍遵循轻量列表优先、详情按需补齐、全量聚合受限的策略，避免为了“智能”牺牲整体可用性。
- 右侧执行过程和审计记录展示工程摘要；普通回答正文保持业务语言，不输出冗长执行日志。

### 15.5 流式内容的页面滚动跟随

- 对话页使用浏览器页面滚动，不在消息容器内创建第二套纵向滚动区域。
- 前端通过 `ResizeObserver` 监听整个消息容器的尺寸变化，统一覆盖思考步骤、执行结果、`answer_delta`、图表渲染和最终消息替换，避免只对某类 SSE 事件做滚动处理。
- 高频增量更新通过 `requestAnimationFrame` 合并为每帧最多一次滚动，并使用即时滚动，避免连续平滑动画排队造成页面落后于内容。
- 提交新问题时重置当前滚动位置基线并默认开启自动跟随，避免从长会话切换到空会话时浏览器的延迟位置修正被误判成用户上滚。用户向上滚动即暂停跟随；向下回到页面底部 160px 范围内后自动恢复。
- 组件卸载时必须移除页面滚动监听、断开 `ResizeObserver` 并取消待执行动画帧，防止重复监听和资源泄漏。

### 15.6 执行中止与取消传播

- 每次流式请求由前端生成唯一 `executionId`，随 `SendMessageRequest` 发送。中止操作以该 ID 为主键，不依赖规划完成后才产生的 `agentRunId`。
- 前端使用独立 `AbortController` 控制当前 SSE 请求；发送按钮在执行期间原位切换为中止按钮。只有显式点击按钮才调用 `/api/conversations/executions/{executionId}/cancel`；组件卸载只关闭本地流，不发送用户取消命令。主动中止不得包装为网络错误。
- 后端使用 `ConversationExecutionRegistry` 管理执行句柄、取消标记和工作线程。执行状态采用 `running -> cancel_requested -> cancelled` 与 `running -> committing -> completed` 两条互斥路径；取消和最终提交通过 CAS 裁决，不能同时成功。只有显式取消接口可以把执行置为 `cancel_requested`，`SseEmitter` 的完成、超时或连接错误不得写入“用户取消”。取消接口返回真实的 `cancelled/state`，前端只有在后端确认取消成功后才关闭本地流。
- SSE 推送连接状态与业务执行取消状态必须分离。页面刷新或网络断开后只把 `streamOpen` 置为 false，后续事件停止写入该连接，但不得调用执行取消；已经提交为 `text/event-stream` 的响应发生 I/O 断开时由异常处理器直接收尾，不得再次包装 JSON 错误响应。
- SSE 任务窗口通过 `cpclaw.conversation.stream-timeout-ms` 配置，默认 600,000ms，覆盖真实跨对象和全量分页分析；Servlet 异步超时由事件流异常处理器直接收尾，不能再进入普通 JSON 全局异常响应。前端仍以收到 `final` 为成功标准，收到 `error` 时展示真实错误，不能把半截正文视为完成。
- 云枢运行态查询使用按主机、账号和凭据指纹隔离的认证会话缓存，成功会话默认复用 10 分钟。跨对象报告中的项目、项目验收等多个查询共享同一认证会话，避免每个对象重复探测 RSA Key 和多种登录端点；缓存键不得保存明文密码。
- `AgentProgressListener` 提供协作式取消检查。规划、元数据匹配、云枢查询、报告分析和回答流在下一可取消边界检查标记；对阻塞中的外部 HTTP 调用同时发送线程中断，返回后不得继续后续步骤。
- 用户问题与取消终态需要持久化。取消后保存助手消息“本次执行已由用户中止，未生成最终结论”，并保留已发生的执行时间线；模型或规则已经产生但尚未完成校验的部分正文不作为业务结论持久化。
- 取消和正常完成存在竞争时，以后端执行句柄的原子终态为准：已取消任务不得再发送 `final`，已完成并发送 `final` 的任务不再展示中止按钮。重复取消不得产生重复消息或重复副作用。

### 15.7 回答尾部交互与用量信息

- 助手消息尾部工具栏复用消息 `metadataJson.usage`，按单条消息展示真实 `completion_tokens` 与 `total_tokens`；不按字符数估算。底部只保留复制按钮和紧随其后的 Token 内联文字，不设置边框、背景或独立胶囊；无 usage 时不展示 Token，中止消息改为展示“已中止”。
- 复制按钮复用现有剪贴板逻辑；同一消息内的 `<details><summary>` 整行是思考与执行过程的唯一展开入口，底部不再提供重复的展开/收起按钮，也不创建新的右侧面板或重复轨迹数据。
- 工具栏始终位于正文、图表和追问之后，采用低对比常显样式；按钮提供 tooltip 与 `aria-label`，Token 使用等宽数字的内联文字，移动端允许工具栏弹性收缩但不得产生水平溢出。

### 15.4 Answer Plan 回答计划层

截图中“哪些商机金额比较高”返回普通列表摘要，根因不是缺少一个金额排行规则，而是系统缺少统一的回答计划层。对象匹配只能回答“查哪个云枢对象”，不能回答“围绕什么指标、按什么维度、用什么排序、以什么形态回答”。因此 CPClaw 在运行态查询前需要生成 `RuntimeQuestionPlan`，作为自然语言问题到执行策略之间的中间语义层。

回答计划至少包含：

- `operation`：回答动作，例如默认查询、聚合、排行、分布、详情、分析。
- `metric`：指标，例如金额、数量、合同额、收入、概率。
- `dimension`：维度，例如阶段、负责人、客户、年份、地区。
- `filters`：筛选条件，例如进行中、本月、某负责人。
- `sort`：排序方向和排序字段，例如按金额降序。
- `limit`：返回范围，例如 Top 10、第一条、前 5 条。
- `responseShape`：回答形态，例如结论优先、摘要列表、分析结论、需要澄清。

当前先在运行态服务内落地最小可用版本：将“指标 + 排名/最高/比较高/Top/前几/哪些高”等表达解析为 `operation=RANKING`，将“金额/合同额/收入/项目金额/商机金额”等表达解析为 `metric=AMOUNT`，再由统一排行执行器按计划抽取指标字段、排序、组织结论和 Top 列表。这样“哪些商机金额比较高”“哪些项目合同额最高”“收入 Top 10 是哪些”都应走同一类 Answer Plan，而不是各自新增孤立分支。

后续增强方向是把 `RuntimeQuestionPlan` 从运行态服务中独立出来，与模型规划器、元数据字段线索和接口元数据共同生成最终执行计划，使回答策略成为可测试、可审计、可扩展的一等对象。

## 16. 元数据驱动的三段式推理执行路径

CPClaw 的推理执行路径必须按三段式闭环运行，而不是只靠规则匹配或只靠大模型自由发挥。

### 16.1 第一段：从用户输入检索本地云枢元数据

系统接收用户输入后，先从本地元数据库检索相关云枢元数据，候选范围包括：

- 应用与应用编码 `AppCode`。
- 实体对象与真实 `SchemaCode`。
- 实体对象下的数据项，包括字段名称、字段编码、字段类型、引用关系和说明。
- 实体对象之间的关联关系，尤其是通过关联表单字段形成的关系路径。
- 业务对象 API 动作，包括集合查询、详情查询、新增、更新、删除等能力及其 HTTP Method、路径、风险等级和是否需要确认。

这一段的输出不是最终答案，而是可提供给模型推理的元数据上下文。任何可执行对象和字段都必须来自已同步的真实云枢元数据，不能由模型猜测生成。

### 16.2 第二段：用户输入 + 云枢元数据组装 Prompt，调用大模型规划执行路径

系统将用户原始输入、会话上下文、候选实体、数据项、关联关系、API 动作和确定性规则识别结果组装为新的 Prompt，调用大模型进行结构化推理，输出执行路径。

执行路径至少应包含：

- 用户真实意图：查询、统计、分析、新增、修改、删除或澄清。
- 目标业务对象：应用、实体名称、`AppCode`、`SchemaCode`。
- 需要使用的数据项：例如负责人字段、金额字段、阶段字段、创建时间字段。
- 过滤条件：例如负责人等于张三、本月、进行中、某客户。
- 指标与维度：例如数量、金额汇总、按年、按阶段、按负责人。
- API 动作：集合查询、详情查询、写操作或删除操作。
- 风险策略：查询类直接执行，新增、更新、删除类必须用户确认后执行。

模型规划结果必须再经过元数据校验：模型只能在已提供的元数据中选择对象、字段和 API 动作，不能输出不存在的 `schemaCode`、字段编码或接口路径。

### 16.3 第三段：调用云枢 API 后，再让大模型基于查询结果总结输出

系统按执行路径调用云枢运行态 API 获取或操作数据。查询类任务获取数据后，不能直接把表结构或原始 JSON 返回给用户，而应将以下内容再次组装给大模型进行总结：

- 用户原始输入和有效问题。
- 命中的云枢应用、实体、数据项、关联关系和 API 动作。
- 实际执行的过滤条件、指标字段和运行态接口。
- 云枢 API 返回的结构化结果、总数、样本记录和聚合结果。
- 规则层已经计算出的确定性结论，例如数量、金额合计、Top 排序、阶段分布。

大模型的职责是把这些真实数据总结成人类可理解的业务答案，并确保回答和用户问题对应。模型不能编造未返回的数据，不能忽略过滤条件，也不能把字段结构直接当成最终答案输出。

当前代码落地要求：

- `AgentOrchestrator` 在运行态查询前生成 `runtimeReasoningContext`，包含用户输入、实体、数据项、关联关系、API 动作、结构化过滤条件、指标字段和模型规划结果。
- `CloudPivotRuntimeService` 在调用模型总结时传入 `reasoningContext`。
- `ModelGateway.analyzeRecords(...)` 支持接收 `reasoningContext`，并在总结 Prompt 中附加元数据与执行路径 JSON。
- 模型不可用或调用失败时，规则层仍提供稳定兜底答案，但兜底答案也必须尊重结构化过滤条件和真实元数据字段。

## 17. 对话 UI 结构调整

对话页采用两栏工作台，而不是三栏结构：

- 页面只包含左侧侧栏和右侧对话主区两个一级区域，不再渲染顶部全局导航和右侧执行详情栏。
- 左侧侧栏承担系统 Logo、侧栏折叠、新建对话、历史会话、用户身份和系统设置入口；历史会话支持选择、新建、删除和移动端抽屉式开合。
- 历史会话列表以 `/api/conversations` 返回的 `updatedAt DESC` 顺序为权威顺序。打开已有会话时只能按原索引替换详情摘要，不能使用 prepend/upsert 把选中项移动到顶部；只有新增会话可前插，消息成功提交后通过重新加载列表应用新的更新时间排序。
- 右侧顶部只承担当前会话名称、Token 消耗和深浅主题切换。模型选择与思考模式不得重复出现在顶部。
- 右侧主体承担消息展示、思考过程、正式回答、洞察报告和输入提交。
- 右侧独立执行详情栏取消，避免挤占对话内容宽度；每轮回复的执行过程统一沉淀到助手消息内部的“思考与执行过程”折叠块。
- 消息内过程块、审计元数据和历史恢复继续复用同一份 `executionTimeline` / `AgentResponse.steps` 数据，不能拆成两套展示逻辑。
- 输入框固定在对话主区底部，桌面端居中宽度约 800px，移动端随屏幕自适应；消息区底部保留足够留白，避免最后一条回答被输入框遮挡。
- UI v2 确认前以 `design/chat-ui-v2-prototype.html` 为设计基准；实现阶段不得直接套用截图中的临时视觉状态，必须以确认后的原型结构、间距、宽度和响应式规则为准。
- UI v2 原型在右侧顶部保留会话名称、Token 消耗与主题切换；主题默认浅色，支持深色并使用本地偏好或用户设置持久化。
- 模型选择、思考深度、附件上传和发送统一放在输入框内部，必须接入现有模型配置、思考开关和附件上传逻辑；输入框内部不再提供新建会话入口。
- 真实实现时，元数据、审计、账号与模型等路由入口统一收纳到左侧系统设置或其二级导航中；用户身份和系统设置入口固定在左侧栏底部。
- 布局尺寸使用共享 CSS 变量约束：左侧头部与右侧会话头部统一读取 `--header-height`，当前确认值为 56px，禁止分别设置 `min-height` 导致分割线错位；左侧用户身份区使用独立的 60px 紧凑高度。
- UI v2 原型使用 URL hash 区分空状态和对话内容状态：无 hash 默认为多轮对话确认状态，`#empty` 为初始化空状态。该机制只用于设计确认，正式 Vue 实现必须由会话消息数量和加载状态驱动，不得依赖 hash。
- 助手消息采用开放内容流而非整段大气泡。完成后的执行过程默认折叠在回答上方，并保留可展开的步骤明细；结构化指标、阶段条形图、业务发现和关联追问作为回答内部组件呈现。多轮追问必须从会话上下文恢复上一轮实体、时间范围与负责人范围。
- 正式实现采用路由级布局分支：`MainLayout.vue` 在聊天路由不渲染全局 Header，并移除外层 padding；元数据、审计和设置路由继续使用现有导航。`ChatView.vue` 保留原有会话、SSE、执行时间线、确认操作和历史恢复逻辑，只重组模板与视觉层。
- 主题状态保存在浏览器本地存储，默认浅色；聊天工作台通过 CSS 变量向模型选择、思考选择、附件、Markdown、执行时间线和洞察报告传递深浅主题。移动端小于等于 900px 时侧栏默认收起并以抽屉方式覆盖主区。
- Token usage 由后端统一采集：`OpenAiCompatibleModelGateway` 从规划调用和总结流式调用的服务商响应中读取 `prompt_tokens`、`completion_tokens`、`total_tokens`，同时兼容 `input_tokens`、`output_tokens`；流式请求通过 `stream_options.include_usage=true` 请求末尾 usage 事件。`ModelUsageContext` 在单次消息处理范围内累加各次模型调用，`ConversationService` 将最终结果写入助手消息 `metadataJson.usage`，没有模型调用也显式写入 0，避免“无数据”和“零消耗”混淆。
- 前端按当前会话全部助手消息的 `metadataJson.usage` 汇总，顶部显示格式化后的总 Token，提示层展示输入、输出和合计。历史消息没有 usage 字段时显示 `Token --`，不得估算或回填；模型配置尚未提供上下文窗口上限，因此当前只展示真实消耗，不展示使用比例。当前后端思考契约仍是布尔值，正式 UI 的“快速回答 / 深度思考”分别映射 `false / true`，不得扩展成没有执行差异的伪多档选项。
- 输入工具栏的模型和思考选择器继续使用 Element Plus `el-select` 以保持键盘与无障碍能力，但通过共享 CSS 变量收敛为 28px 高度，并放入 36px 的统一控制组。附件自动上传按钮和发送按钮统一为 32px；发送状态由页面显式渲染 15px `Loading` 图标，避免 `el-button :loading` 注入的默认加载结构破坏按钮尺寸。
- 助手和系统消息的 `.message__body` 必须显式占用消息容器扣除 30px 头像与 11px 间距后的完整宽度，即 `calc(100% - 41px)`；不能只使用 `max-width`，否则短回答会触发 flex 内容收缩，导致同一会话的思考过程条宽度不一致。执行中的 `.pending-thinking` 和完成后的 `.message-thinking-summary` 还必须显式使用 `width: 100%` 与 `box-sizing: border-box`，保证运行态、完成态和最终回答严格同宽。用户消息不应用该固定宽度规则。
