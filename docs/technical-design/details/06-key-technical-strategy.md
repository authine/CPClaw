# CPClaw 关键技术策略

## 1. 配置与元数据必须持久化

CPClaw 的系统配置、账号配置、模型配置和元数据初始化结果都必须持久化到数据库中，不能只保存在前端状态、内存、临时会话或 H2 临时库里。

必须持久化的内容包括：

- 管理员元数据环境配置：云枢设计态/管理端地址、管理员账号、管理员凭据引用、搜索引擎配置等。
- 个人云枢账号配置：云枢运行态地址、用户账号、用户凭据引用等。
- 模型配置：模型名称、模型 API 地址、模型 API Key 凭据引用、模型显示名、是否支持思考模式、默认思考模式等。
- 云枢元数据初始化结果：应用、模型、表单、字段、流程、Action、关系、检索文档、同步批次、同步时间等。
- 会话、消息、Agent 执行记录、确认记录和审计记录。

凭据类信息只能加密保存，数据库中保存凭据引用或密文，不允许在日志、文档、前端响应或 Git 中出现明文密码、Token、API Key。

当前使用 H2 内存库只能作为临时验证环境。生产和正式联调必须使用持久化数据库，例如 MySQL。重启服务后，账号配置、模型配置、元数据索引和历史记录都应该仍然存在。

## 2. 核心技术主线

CPClaw 的核心逻辑不是普通聊天，也不是让用户自己说明云枢对象，而是：

```text
用户自然语言
  -> 理解用户意图
  -> 从已同步的云枢元数据中匹配应用、模型、表单、字段或 Action
  -> 将意图拆解为可执行任务
  -> 调用云枢 API 查询或操作运行态数据
  -> 将真实数据交给规则聚合或大模型推理总结
  -> 返回结果、依据和执行过程
```

云枢元数据是 CPClaw 理解用户意图和生成执行计划的知识底座。Agent 必须基于已同步的真实云枢元数据进行对象识别、能力定位和任务编排，不能凭空猜 `schemaCode`，也不能使用本地演示数据冒充真实业务数据。

## 3. 元数据匹配的非向量检索方案

当前阶段先不做向量化语义检索。优先实现确定性强、可解释、可测试的元数据检索能力：

1. 精确名称召回：模型、应用、表单或字段名称与业务对象词精确一致，例如“商机”“客户”“线索”。
2. 别名召回：将业务别名映射到标准业务对象，例如“销售机会/机会/opportunity”映射为“商机”，“潜客/lead”映射为“线索”。
3. 编码召回：用户直接输入或上下文中出现 `schemaCode`、应用编码、字段编码时，优先按编码召回。
4. 应用路径召回：使用 `graphPath` 匹配应用和对象路径，例如 `CRM / 商机`、`CRM / 客户`。
5. 全文关键词召回：从 Metadata Index 的 `searchText` 中召回名称、编码、应用描述、模型描述和操作关键词。
6. 复合业务词召回：对“客户下面的商机”“商机跟进记录”等多对象表达，识别最终目标对象和相邻业务对象，优先召回可执行目标对象。
7. 辅助对象降权：对“管理、分配、变更、转移、统计、报表、查询、测试、test”等辅助表或测试表降权，避免误选非核心对象。

向量检索保留为后续增强项。当前实现不得依赖向量库、Embedding 或不可解释语义相似度来决定核心业务对象。

## 4. 元数据匹配流程

意图识别后的元数据匹配流程为：

```text
结构化意图
  -> 业务词清洗
  -> 非向量多路召回
  -> 统一打分排序
  -> 图谱/路径校验
  -> 字段和动作补全
  -> 置信度判断
  -> 输出可执行任务
```

业务词清洗需要去掉“系统、有多少、情况怎么样、全部、一下、数据、信息”等非对象词，保留真实业务对象词。例如：

- “系统有多少商机？”保留“商机”。
- “每年的客户量情况怎么样？”保留“客户”，并提取维度“年份”。
- “把系统中的线索全部分析一下”保留“线索”，并识别动作为分析。

召回阶段宁可多拿候选，也不能过早截断。排序阶段再综合名称、编码、应用路径、对象类型、别名、辅助对象降权和上下文目标对象进行打分。

## 5. 查询类任务

当用户表达的是查询、统计、汇总、分析、洞察、趋势判断等读数据意图时，如果意图和对象已经明确，CPClaw 应直接执行，不应要求用户重复补充云枢对象。

示例：

```text
用户：系统有多少商机？
```

CPClaw 应理解为：

- 意图：查询/统计。
- 业务对象：商机。
- 执行动作：统计商机数量。
- 元数据匹配：从云枢元数据中找到商机对象。
- API 动作：调用云枢运行态查询接口，获取商机总数。
- 返回结果：商机总数，并展示命中的对象、真实 `schemaCode`、运行态 total、原始数据摘要和结论生成过程。

示例：

```text
用户：把系统中的线索全部分析一下。
```

CPClaw 应理解为：

- 意图：分析。
- 业务对象：线索。
- 执行动作：查询线索数据并分析。
- 元数据匹配：从云枢元数据中找到线索对象。
- API 动作：调用云枢运行态接口获取线索数据。
- 分析动作：将线索数据交给大模型或规则分析模块。
- 返回结果：线索分析结论、关键发现、风险信号和下一步建议。

## 6. 操作类任务

当用户表达的是新增、修改、删除、提交、作废、流程审批、执行 Action、写入跟进记录等会改变云枢数据的意图时，CPClaw 不能直接执行。

操作类任务必须经过以下流程：

```text
理解用户意图
  -> 匹配云枢元数据对象
  -> 生成待执行操作计划
  -> 展示影响范围和风险
  -> 请求用户确认
  -> 用户确认后再调用云枢 API 执行
  -> 记录审计
  -> 返回执行结果
```

例如“给第一条商机写一条跟进记录”，系统需要先确认上一轮第一条记录是否能绑定到具体记录 ID、跟进记录对象或 Action 是否存在、跟进内容是否完整，以及新增操作是否需要用户确认。

## 7. 澄清与意图收敛

如果 CPClaw 无法从云枢元数据中准确匹配用户意图，系统应通过多轮对话引导用户补充信息，直到获得更真实、更准确、可执行的意图。

触发澄清的典型情况包括：

- 用户表达过于模糊，例如“帮我处理一下”“看一下数据”。
- 用户说出了动作，但没有明确业务对象，例如“统计一下数量”。
- 用户说出了对象，但云枢元数据中存在多个相近对象，例如多个“线索”“客户”“商机”。
- Top1 候选对象置信度不够高，或 Top1/Top2 分数接近，直接执行可能查错对象。
- 用户使用了业务简称、口语化表达或公司内部叫法，系统无法稳定映射到云枢模型。
- 用户要求写入、修改、删除、提交等操作，但缺少必要字段或上下文记录。
- 用户引用“第一条”“刚才那个”“这个客户”等上下文，但系统无法绑定到具体记录 ID。

澄清时不能只回复“我没有理解你的云枢对象”，而应该说明：已理解到的内容、当前不确定的地方、从云枢元数据中找到的可能相关对象，以及用户只需要补充哪一个关键信息。

核心原则：能准确匹配时直接执行；不能准确匹配时主动澄清；澄清要基于已同步的云枢元数据给出候选；用户补充后继续执行，不要让任务断掉。

## 8. 思考过程与流式输出

CPClaw 的对话交互应采用流式输出体验，参考 DeepSeek 的交互方式，让用户能看到系统正在理解、检索、执行和总结，而不是长时间等待后一次性返回结果。

流式输出阶段包括：

```text
接收用户问题
  -> 输出正在理解意图
  -> 输出正在匹配云枢元数据
  -> 输出正在生成执行计划
  -> 输出正在调用云枢 API
  -> 输出正在分析真实数据
  -> 流式输出最终回答
```

思考过程展示的是可审计的工程化摘要，不是暴露不可控的内部隐式推理原文。展示内容应包括意图理解、业务对象识别、元数据匹配、执行动作、API 调用过程、数据结果摘要、结果生成方式和 Reflection 检查。

## 9. 大模型的角色

大模型不是用来凭空猜数据，而是用于：

- 理解用户自然语言意图。
- 辅助识别业务对象、动作、维度和筛选条件。
- 将云枢元数据转化为可执行计划。
- 对真实查询结果进行总结、分析、归纳和建议生成。
- 在信息不足时生成更好的澄清问题。

大模型不能替代真实云枢查询。业务数据必须来自云枢 API。

## 10. 总结原则

CPClaw 的关键技术原则可以总结为：

```text
配置要持久化。
元数据是知识底座。
当前先做非向量的确定性检索：精确名称、别名、编码、应用路径、全文关键词和复合业务词召回。
意图要转换为云枢对象和动作。
查询类任务直接执行并返回真实结果。
操作类任务必须确认后执行。
意图不明确或元数据匹配不稳定时，通过对话引导用户补充信息。
所有业务结论必须来自真实云枢运行态数据。
大模型负责理解、规划和总结，不能编造业务事实。
交互上必须支持流式输出和可审计执行过程。
```
